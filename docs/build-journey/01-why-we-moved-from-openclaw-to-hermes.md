# 왜 우리는 OpenClaw에서 Hermes로 넘어왔나

## TL;DR
OpenClaw에서 Hermes로 넘어온 건 이름만 바꾼 일이 아니었습니다. 핵심은 더 멋진 에이전트 페르소나를 만드는 게 아니라, **에이전트를 실제로 운영 가능한 시스템으로 다시 세우는 것**이었습니다. 메인 창구, 역할 분리, 프로필 경계, 기억 레이어, 도구 연결, 기록 자산화까지 한 번에 보려면 OpenClaw 감각만으로는 부족했고, Hermes라는 더 선명한 운영 기준점이 필요했습니다.

![왜 우리는 OpenClaw에서 Hermes로 넘어왔나](/assets/build-journey/why-we-moved-from-openclaw-to-hermes/og-card-final.png)

## 이 글의 답
먼저 답부터 말하면 이렇습니다.

우리가 OpenClaw에서 Hermes로 넘어온 이유는 단순했습니다.
**잘 말하는 에이전트보다, 실제로 굴러가는 에이전트 시스템이 더 필요해졌기 때문**입니다.

조금 더 풀면 세 가지입니다.
- 사용자가 여러 에이전트를 직접 관리하지 않게 만들 **메인 창구**가 필요했습니다.
- session, memory, skill, profile, identity를 한 덩어리 감각이 아니라 **운영 가능한 층**으로 봐야 했습니다.
- 대화에서 끝나지 않고, 도구 실행과 Obsidian 기록, 블로그/강의 자산화까지 이어지는 **운영 흐름**이 필요했습니다.

그래서 Hermes는 OpenClaw의 부정이 아니라, **그 유산 위에서 현재 운영 기준을 더 선명하게 만든 다음 단계**에 가깝습니다.

---

## 많은 전환이 실패하는 이유
에이전트 시스템을 갈아탈 때 사람들은 보통 이렇게 생각합니다.

- 이름 바꾸면 되겠지
- 프롬프트만 조금 손보면 되겠지
- 기존 감각을 새 시스템에 그대로 옮기면 되겠지

그런데 실제 운영은 늘 다른 문제를 꺼냅니다.

- 사용자는 누구를 먼저 불러야 하는지 고민하고 싶지 않습니다.
- 좋은 대화는 쌓이는데 다시 꺼내 쓰기는 어렵습니다.
- 반복 작업은 계속 생기는데 절차는 재사용되지 않습니다.
- 도구를 붙이면 설정, 인증, 프로세스, 기록이 서로 얽힙니다.
- 과거 시스템 흔적과 현재 시스템 기준이 섞이면 체감 혼선이 커집니다.

즉 문제는 "에이전트가 똑똑한가"보다 **운영 구조가 얼마나 선명한가**로 이동합니다.

OpenClaw는 이 문제를 발견하게 만든 중요한 출발점이었고,
Hermes는 그 문제를 더 노골적으로 해결하려는 체계였습니다.

---

## OpenClaw에서 Hermes로 넘어오며 실제로 바뀐 것

![이름 변경이 아니라 구조 전환](/assets/build-journey/why-we-moved-from-openclaw-to-hermes/article-figure-01-transition-structure.png)

### 1. 멀티 에이전트보다 단일 창구가 중요해졌다
실무에서는 에이전트가 여러 명 있는 것보다,
**사용자가 누구를 어떻게 불러야 하는지 고민하지 않아도 되는 구조**가 더 중요했습니다.

그래서 Hermes에서는 하비가 메인 창구가 됩니다.
- 하비: 판단, 통합, 실행 책임
- 방울이: 조사
- 뽀동이: 정리

겉으로는 한 창구처럼 보이지만,
안에서는 역할이 나뉘어 움직입니다.

이 차이는 생각보다 큽니다.
OpenClaw 감각이 "개별 에이전트와의 상호작용"에 더 가까웠다면,
Hermes는 **사용자 경험은 단순하게, 내부 역할은 더 선명하게** 가져갑니다.

### 2. 기억을 한 덩어리로 보지 않게 됐다
운영이 길어질수록 "기억"은 하나가 아니었습니다.
실제로는 아래를 분리해서 봐야 했습니다.

- session: 과거 대화와 작업 이력
- memory: 장기 선호와 규칙
- skill: 재사용 가능한 절차
- AGENTS/SOUL: 역할과 정체성
- profile: 어떤 상태 저장 단위를 보고 있는가

이걸 하나로 뭉개서 보면,
"왜 같은 하비인데 오늘은 다르게 느껴지지?" 같은 질문이 끝없이 나옵니다.

Hermes는 이걸 감각이 아니라 **경계와 저장소의 문제**로 번역하기 더 좋은 체계였습니다.

### 3. 대화보다 운영 흐름이 중요해졌다
Hermes에서 중요한 건 예쁜 답변 하나가 아닙니다.

- 요청을 받는다
- 해석한다
- 필요하면 역할을 태운다
- 실제로 도구로 확인한다
- 결과를 정리한다
- Obsidian에 남긴다
- 이후 블로그, FAQ, 강의로 다시 쓴다

즉 Hermes는 챗봇이라기보다,
**오케스트레이터 + 실행 시스템 + 기록 시스템**에 가깝습니다.

---

## 겉으로는 단순 교체처럼 보였지만, 실제 전환은 훨씬 복잡했다

![체감 변화는 경계 문제로 번역된다](/assets/build-journey/why-we-moved-from-openclaw-to-hermes/article-figure-02-boundary-translation.png)

여기서 중요한 포인트가 하나 있습니다.

OpenClaw에서 Hermes로 넘어오는 일은
"옛날 이름 지우고 새 이름 붙이기"처럼 깔끔하게 끝나지 않습니다.

실제로는 이런 흔적이 남습니다.
- heartbeat 용어 혼선
- LaunchAgent/autostart 잔존
- OpenClaw 관련 comments, imports, skill 흔적
- identity가 한 파일이 아니라 여러 레이어에 흩어져 있는 문제
- `default → harvey` 전환 후 체감 차이

이건 실패한 전환이라서가 아닙니다.
오히려 **실제 시스템 전환이 원래 이렇게 덜 깔끔하다**는 걸 보여줍니다.

### 예를 들어 이런 혼선이 생깁니다
- "왜 아직 OpenClaw가 살아 있는 것처럼 보이지?"
- "왜 heartbeat가 남아 있지?"
- "왜 하비가 예전이랑 다른 사람처럼 느껴지지?"
- "왜 예전 기억이 약해진 것 같지?"

사용자 입장에서는 다 감각의 변화처럼 보이지만,
운영자 입장에서는 아래 문제로 번역됩니다.
- 어떤 profile이 활성인가
- 어떤 state DB를 보고 있나
- 어떤 skills가 설치돼 있나
- 어떤 identity docs를 읽고 있나
- 어떤 runtime이 실제로 살아 있나

이 차이를 설명할 수 있어야 전환이 안정됩니다.

---

## 그래서 OpenClaw를 버린 게 아니라, 위치를 다시 정리한 것이다

![과거를 지우는 게 아니라 기준점을 다시 세우는 일](/assets/build-journey/why-we-moved-from-openclaw-to-hermes/article-figure-03-legacy-governance.png)

여기서 자주 나오는 오해가 있습니다.

**"그럼 OpenClaw는 실패고 Hermes가 정답인가?"**

그렇게 보지는 않습니다.
오히려 더 정확한 설명은 이렇습니다.

- OpenClaw는 중요한 출발점이었습니다.
- 여러 역할 감각과 운영 실험의 기반이 됐습니다.
- Hermes는 그 유산 위에서, 현재 운영에 맞게 구조와 기준을 더 선명하게 만든 체계입니다.

즉 전환의 핵심은 단절이 아니라,
**유산의 재구성**입니다.

과거 흔적을 완전히 지우는 게 목표는 아닙니다.
다만 그 흔적이 현재 active 문서와 current source of truth를 흐리지 않게 해야 합니다.

이 기준이 없으면 이런 문제가 반복됩니다.
- 과거 설명이 현재 규칙처럼 보임
- OpenClaw 흔적이 active doc로 튀어나옴
- 현재 무엇을 기준으로 봐야 하는지 애매해짐

그래서 정답은 삭제가 아니라 **거버넌스와 위치 정리**입니다.

## OpenClaw와 Hermes를 한 표로 보면
| 항목 | OpenClaw 쪽 인상 | Hermes 쪽 인상 |
|---|---|---|
| 중심 감각 | 이전 유산/캐릭터 흔적 | 현재 운영 구조 |
| 사용자 창구 | 다소 맥락 의존적 | 하비 중심 단일 창구 |
| 역할 분리 | 상대적으로 흐릴 수 있음 | 조사/정리/총괄 분리 명확 |
| 프로필 인식 | 덜 노출될 수 있음 | 구조적으로 중요하게 다룸 |
| 기억 체계 | 혼재되어 보일 수 있음 | session/memory/skill 분리 |
| 도구 사용 | 대화 중심 인상 | 검증/실행 중심 강조 |
| 문서화 | 흔적 중심 | 위키/자산화 지향 |
| 시스템 설명 방식 | 캐릭터/맥락 감각 | 운영 구조 감각 |

이 표에서 중요한 건 우열이 아닙니다.
**무엇을 현재 기준점으로 삼을 것인가**입니다.

## 우리가 실제로 배운 운영 원칙
전환을 겪고 나서 남은 건 기술 팁만이 아니었습니다.
운영 원칙도 같이 생겼습니다.

### 원칙 1. 이름보다 실행 주체를 먼저 본다
heartbeat가 보여도, 예전 단어가 남아 있어도,
실제 프로세스와 runtime을 먼저 확인해야 합니다.

### 원칙 2. 체감 변화는 저장소 경계로 번역한다
"하비가 달라졌다"는 말은 종종 맞는 체감입니다.
하지만 운영적으로는 다음 질문으로 바꿔야 합니다.
- profile이 달라졌는가
- state DB가 달라졌는가
- skill 구성이 달라졌는가
- identity 문서가 달라졌는가

### 원칙 3. 과거는 지우지 말고 위치를 옮긴다
OpenClaw 유산은 archive, history, comparison, migration note 쪽에 남길 수 있습니다.
대신 현재 운영 기준 문서에서는 Hermes가 먼저 보여야 합니다.

### 원칙 4. 전환사는 좋은 콘텐츠 소재다
많은 에이전트 시스템이 비슷한 과정을 겪습니다.
그래서 이 전환사는 내부 회고에만 머물지 않고,
다른 팀에게도 꽤 좋은 학습 소재가 됩니다.

---

## 결론
OpenClaw에서 Hermes로 넘어온 건 더 화려한 이름을 찾은 일이 아니었습니다.

핵심은 이것입니다.
**에이전트를 말 잘하는 존재로 두는 대신, 실제로 굴러가는 운영 시스템으로 다시 세우는 일.**

그리고 그 과정에서 우리는 배웠습니다.
- 좋은 에이전트는 답변만 잘하면 안 된다.
- 메인 창구와 역할 분리가 필요하다.
- 기억은 한 덩어리가 아니라 레이어다.
- 도구와 기록이 이어져야 운영이 된다.
- 과거 유산은 지우지 말고, 현재 기준점만 분명히 세워야 한다.

그래서 Hermes는 단순 리브랜딩이 아니라,
**운영 구조를 다시 세운 다음 단계**였습니다.

## FAQ
### 1. OpenClaw와 Hermes는 완전히 다른 시스템인가요?
완전히 단절된 시스템이라기보다, OpenClaw의 유산 위에서 Hermes가 현재 운영 기준을 더 선명하게 세운 체계에 가깝습니다.

### 2. 왜 전환 후에도 OpenClaw 흔적이 보였나요?
실제 시스템 전환은 문서, autostart, skill, comments, runtime 흔적이 모두 한 번에 정리되지 않기 때문입니다. 이상한 일이 아니라 자연스러운 전환 후유증에 가깝습니다.

### 3. 왜 같은 하비인데 다르게 느껴졌나요?
보통은 profile, state DB, skill, identity docs 같은 상태 저장 단위가 달라졌기 때문입니다. 감정 문제가 아니라 경계 문제인 경우가 많습니다.

### 4. OpenClaw 흔적은 다 지워야 하나요?
아닙니다. archive/history/comparison 문서에는 남길 수 있습니다. 다만 현재 운영 기준 문서에는 Hermes 기준이 먼저 보여야 합니다.

### 5. 이 전환이 다른 팀에게도 의미가 있나요?
있습니다. 많은 에이전트 시스템이 결국 "좋은 답변"에서 "운영 가능한 구조"로 이동하기 때문입니다. 이 전환사는 그런 변화의 실제 사례로 쓸 수 있습니다.
