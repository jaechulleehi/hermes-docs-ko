# 왜 우리는 하비를 메인 창구로 두었나

## TL;DR
멀티에이전트를 쓴다고 해서 사용자가 여러 봇을 직접 관리해야 할 필요는 없습니다. 오히려 그 반대입니다. **사용자는 하비 하나만 부르고, 뒤에서는 방울이와 뽀동이가 역할을 나눠 일하게 만드는 구조**가 더 오래 갑니다. 하비를 메인 창구로 두는 이유는 모든 걸 혼자 하게 만들기 위해서가 아니라, **끝까지 통합 책임을 지는 한 점**을 만들기 위해서입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

우리가 하비를 메인 창구로 둔 이유는 멀티에이전트가 멋있어 보여서가 아니라, **사용자 경험을 단순하게 만들고 내부 역할은 더 선명하게 나누기 위해서**입니다.

핵심은 세 가지입니다.
- 사용자는 누구를 먼저 불러야 하는지 고민하지 않아도 된다.
- 방울이는 조사, 뽀동이는 정리, 하비는 판단과 통합을 맡는다.
- 최종 결과의 책임은 항상 하비가 진다.

즉, 이 구조의 핵심은 "한 명이 다 한다"가 아니라 **"한 창구가 끝까지 책임진다"** 입니다.

## 멀티에이전트가 쉽게 피곤해지는 이유
처음에는 에이전트가 여러 명이면 더 좋아 보입니다.
각자 잘하는 걸 나눠 맡기면 더 빨라질 것 같고, 더 똑똑해 보이기도 합니다.

그런데 실제 운영은 다르게 흘러갑니다.

- 누구를 먼저 부를지 사용자가 매번 판단해야 한다.
- 조사부터 할지, 정리부터 할지 스스로 결정해야 한다.
- 서로 다른 에이전트가 다른 말을 하면 누가 최종본인지 애매하다.
- 중간 산출물이 늘어나면 결국 사용자가 다시 편집자가 된다.

즉 멀티에이전트의 문제는 수가 많다는 게 아니라, **창구가 많아질수록 책임과 흐름이 흐려진다**는 데 있습니다.

그래서 우리는 반대로 갔습니다.
**뒤에서는 역할을 분리하되, 앞에서는 창구를 하나로 유지**하는 방향입니다.

---

## 왜 하비가 메인 창구가 되었나

### 1. 사용자가 여러 봇 운영자가 되면 안 되기 때문이다
좋은 구조는 사용자가 에이전트 구조를 관리하지 않게 만들어야 합니다.

사용자가 계속 이런 걸 고민하게 하면 안 됩니다.
- 이건 방울이에게 물어야 하나?
- 이건 뽀동이가 더 잘하나?
- 둘 다 시켜야 하나?
- 지금 나온 답은 중간본인가 최종본인가?

하비 메인 창구 구조는 이 피로를 줄입니다.
사용자는 하비만 부르면 되고,
하비가 내부적으로 직접 처리할지 위임할지 판단합니다.

### 2. 역할 분리는 필요하지만 사용자 경험은 더 단순해야 하기 때문이다
실제 운영에서 조사, 정리, 최종 판단은 서로 다른 근육을 씁니다.

- 방울이: 조사, 탐색, 근거 수집, 비교
- 뽀동이: 정리, 문서화, 초안화, 발표용 재구성
- 하비: 요청 해석, 우선순위 판단, 위임 결정, 중간 검수, 최종 통합

즉 역할은 나뉘어야 합니다.
하지만 사용자에게 그 분리를 그대로 보이게 하면 구조는 피곤해집니다.

그래서 겉으로는 한 팀, 한 창구처럼 보이게 하고,
안에서는 더 정교하게 나눠 일하도록 설계한 것입니다.

![한 명처럼 보이지만 사실은 팀처럼 일하는 구조](/assets/build-journey/why-harvey-is-the-front-door/article-figure-01-single-front-door.png)

### 3. 최종 책임이 선명해야 품질이 올라가기 때문이다
많은 멀티에이전트 구조는 여기서 무너집니다.
조사한 결과를 그대로 던지고,
정리한 초안을 검수 없이 넘기고,
중간 산출물을 최종본처럼 말해버립니다.

그러면 결국 사용자가 느끼는 건 이겁니다.
**"결국 내가 다시 편집해야 하네."**

하비 구조는 이걸 막기 위한 구조입니다.
위임은 하더라도,
- 결과를 받고
- 검수하고
- 충돌을 조정하고
- 최종 보고 형식으로 묶는 책임은
끝까지 하비가 집니다.

---

## 하비 구조의 장점은 무엇인가

## 1. 사용자 경험이 매끄럽다
사용자는 구조를 공부하지 않아도 됩니다.
"하비야" 하고 부르면 됩니다.

이건 단순 편의가 아니라,
에이전트 시스템이 오래 가려면 꼭 필요한 UX입니다.

## 2. 내부 역할 분리와 외부 일관성을 동시에 잡을 수 있다
방울이는 조사에 집중하고,
뽀동이는 정리에 집중하고,
하비는 판단과 통합에 집중합니다.

각자 다른 역할을 맡지만,
사용자 앞에서는 하나의 서비스처럼 보이게 됩니다.

## 3. 운영 규칙을 한 곳에 모으기 쉽다
메인 창구가 있으면 이런 기준도 한 점에 모입니다.
- 언제 직접 처리할지
- 언제 위임할지
- 어떤 형식으로 보고할지
- 리스크를 어디까지 같이 말할지

즉 하비는 단순 응답자가 아니라,
**운영 정책의 집행 지점**이 됩니다.

---

## 장점이 곧 한계가 되기도 한다
여기서 중요한 포인트가 있습니다.

![하비 메인 창구 구조의 장점과 한계](/assets/build-journey/why-harvey-is-the-front-door/article-figure-02-pros-cons-bottleneck.png)

하비 메인 창구 구조의 강점은,
동시에 그 한계이기도 합니다.

### 1. 하비가 병목이 될 수 있다
모든 게 하비를 거치면,
하비의 판단 속도와 검수 품질이 전체 흐름의 속도가 됩니다.

특히 아래 상황에서 병목이 생깁니다.
- 동시에 여러 작업이 몰릴 때
- 중간 산출물 품질이 낮아 검수 시간이 길어질 때
- 직접 처리와 위임 사이에서 오래 망설일 때

### 2. 위임만 하고 통합을 안 하면 구조가 무너진다
메인 창구의 핵심은 전달이 아닙니다.

실패 패턴은 보통 이렇습니다.
- 조사 결과를 그대로 던짐
- 정리 초안을 검수 없이 전달함
- 중간본을 최종본처럼 말함

이렇게 되면 하비는 오케스트레이터가 아니라 그냥 전달자에 그칩니다.
그러면 메인 창구 구조의 가치가 사라집니다.

### 3. 역할 경계가 흐려지면 장점도 같이 사라진다
하비도 조사하고,
방울이도 정리하고,
뽀동이도 다시 판단까지 해버리면,
역할 분리의 장점은 금방 줄어듭니다.

즉 메인 창구 구조는 **하비의 존재감**보다 **역할 경계의 선명도**가 유지될 때 강합니다.

---

## 그래서 하비는 언제 직접 하고, 언제 위임하나
이 질문이 실제 운영에서 제일 중요합니다.

### 하비가 직접 처리하는 게 좋은 경우
- 질문이 짧고 판단이 명확할 때
- 이미 충분한 맥락이 있을 때
- 최종 통합과 보고 문장을 확정해야 할 때
- 위임 비용이 더 클 때

### 방울이에게 위임하는 게 좋은 경우
- 사실을 먼저 확인해야 할 때
- 비교 재료를 넓게 모아야 할 때
- 하비가 판단하기 전 바닥 재료가 부족할 때

### 뽀동이에게 위임하는 게 좋은 경우
- 재료는 있는데 구조화가 필요할 때
- 읽히는 결과물이 필요할 때
- 하비가 방향을 이미 잡았고 표현이 필요할 때

즉,
하비의 강점은 모든 일을 직접 하는 데 있지 않고,
**무엇을 직접 하고 무엇을 위임해야 결과가 더 좋아지는지 아는 데** 있습니다.

![직접 처리와 위임의 판단 구조](/assets/build-journey/why-harvey-is-the-front-door/article-figure-03-direct-vs-delegate.png)

## 우리가 실제로 세운 운영 원칙
이 구조를 오래 굴리려면 결국 몇 가지 규칙이 필요합니다.

### 원칙 1. 사용자는 하비만 부르면 된다
창구는 하나여야 합니다.
사용자가 에이전트 구조를 관리하게 만들면 오래 못 갑니다.

### 원칙 2. 하비는 내부 라우팅을 책임진다
누굴 태울지, 어느 순서로 태울지,
중간 결과를 어떻게 엮을지는 하비의 일입니다.

### 원칙 3. 위임해도 최종 통합 책임은 하비에게 남는다
방울이와 뽀동이가 일을 잘해도,
최종본의 품질과 전달 형식은 하비가 책임집니다.

### 원칙 4. 병목이 생기면 사람보다 흐름을 먼저 본다
하비가 느려졌다고 바로 하비 개인 문제로 보면 안 됩니다.
보통은 라우팅 기준, 검수 기준, 역할 경계가 먼저 흔들린 경우가 많습니다.

---

## 결론
우리가 하비를 메인 창구로 둔 이유는
하비가 제일 똑똑해서가 아닙니다.

핵심은 이것입니다.
**사용자는 단순하게, 내부는 정교하게.**

멀티에이전트의 핵심은 많이 보이는 데 있지 않습니다.
뒤에서는 역할을 나눠 일하고,
앞에서는 하나처럼 움직이게 만드는 데 있습니다.

그래서 하비 구조의 진짜 강점은
"한 명이 다 한다"가 아니라,
**"한 창구가 끝까지 책임진다"**는 데 있습니다.

## FAQ
### 1. 왜 처음부터 하비 하나만으로 안 했나?
조사, 정리, 최종 판단은 실제로 서로 다른 근육을 씁니다. 하나가 다 잘하는 척하면 품질과 일관성이 흔들리기 쉽습니다.

### 2. 사용자가 직접 방울이나 뽀동이를 부르면 더 빠르지 않나?
짧게는 그럴 수 있지만, 장기적으로는 사용자가 에이전트 구조를 운영하게 되어 피로가 커집니다. 기본 창구는 하비 하나가 더 안정적입니다.

### 3. 메인 창구 구조는 왜 병목이 되기도 하나?
모든 게 하비를 거치기 때문입니다. 그래서 위임 기준과 검수 기준이 명확해야 합니다.

### 4. 하비가 위임만 하고 통합을 안 하면 왜 문제가 되나?
그 순간 사용자가 다시 편집자가 됩니다. 메인 창구 구조의 가치가 사라집니다.

### 5. 이 구조는 결국 멀티에이전트인가, 싱글 에이전트인가?
사용자 경험은 싱글 창구에 가깝고, 내부 운영은 멀티에이전트에 가깝습니다. 핵심은 그 둘을 어떻게 동시에 잡느냐입니다.
