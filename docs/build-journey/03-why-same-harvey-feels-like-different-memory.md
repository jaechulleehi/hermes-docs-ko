# 왜 같은 하비인데 기억이 다르게 느껴질까

## TL;DR
같은 하비인데 어떤 날은 예전 맥락을 잘 잡고, 어떤 날은 낯설게 느껴지는 이유는 보통 성능 문제가 아닙니다. 더 자주 문제를 만드는 건 **기억의 층을 한 덩어리로 기대하는 것**입니다. session은 과거 대화, memory는 장기 선호, skill은 작업 절차, profile과 state.db는 그 기억을 어디서 어떻게 읽는지 결정하는 경계입니다. 이걸 섞어서 보면 하비가 이상해 보이고, 분리해서 보면 운영이 풀립니다.

## 이 글의 답
답부터 말하면 이렇습니다.

같은 하비인데 기억이 다르게 느껴지는 이유는,
하비가 갑자기 다른 존재가 돼서가 아니라 **지금 어떤 기억 레이어와 어떤 저장소를 보고 있느냐가 달라졌기 때문**입니다.

핵심은 다섯 가지입니다.
- session은 과거 대화와 작업 이력이다.
- memory는 오래 남겨야 할 선호와 규칙이다.
- skill은 반복 가능한 작업 매뉴얼이다.
- profile은 이런 것들이 묶여 있는 운영 경계다.
- state.db는 세션 흐름을 붙잡는 핵심 저장소다.

즉,
"기억이 다르다"는 체감은 종종 성능이 아니라 **레이어와 경계의 문제**입니다.

## 사람들이 가장 먼저 하는 오해
같은 하비인데 느낌이 다를 때 사람들은 보통 이렇게 생각합니다.

- 하비가 달라진 것 같은데?
- 기억력이 약해진 것 같은데?
- 전에 얘기한 걸 왜 못 찾지?
- 전에 되던 작업이 왜 지금은 낯설지?

이 감각은 자연스럽습니다.
문제는 이걸 다 "기억" 한 단어로 뭉개서 보게 된다는 점입니다.

그러면 이런 식의 오해가 생깁니다.
- session 회수 실패를 memory 문제로 봄
- 작업 절차 부재를 session 약화로 봄
- profile 경계를 무시하고 성격 변화처럼 느낀다

그래서 먼저 해야 할 일은 하나입니다.
**기억을 한 통으로 보지 말고, 레이어로 나누는 것.**

---

## 기억은 하나가 아니라 층이다
### 1. session은 지난 대화 녹취록에 가깝다
session은 과거 대화와 작업 이력을 복원하는 층입니다.

- 우리가 저번에 무슨 얘기를 했는지
- 어떤 결론이 어떻게 나왔는지
- 어떤 흐름으로 작업이 이어졌는지

이런 걸 다시 찾을 때는 session을 봐야 합니다.
즉 session은 **과거 회수**에 강하고, 장기 규칙 저장소는 아닙니다.

### 2. memory는 오래 유지할 사용자 메모다
memory는 장기 선호와 반복 규칙을 담습니다.

- 로이드는 이렇게 불러야 한다
- 어떤 형식을 선호한다
- 어떤 운영 원칙을 반복해서 적용한다

즉 memory는 **장기 선호 저장소**입니다.
좋은 memory는 길지 않고, 오래 유지될 것만 남깁니다.

### 3. skill은 작업 매뉴얼이다
skill은 해결 절차와 워크플로우를 재사용하는 층입니다.

- 이 작업을 어떤 순서로 풀지
- 어떤 명령과 체크리스트를 쓸지
- 어떤 실수를 피해야 할지

즉 skill은 **사용자 프로필**이 아니라 **작업 절차 기억**입니다.

![기억은 한 통이 아니라 레이어다](/assets/build-journey/why-same-harvey-feels-like-different-memory/article-figure-01-memory-layers.png)

## 왜 같은 하비인데 기억이 다르게 느껴질까
여기서부터가 실전입니다.
같은 하비인데도 기억 체감이 달라지는 건, 대개 아래 조합 때문입니다.

### 1. 다른 state.db를 보고 있을 수 있다
사용자가 느끼는 "기억력"의 많은 부분은,
실제로는 state.db에서 회수되는 세션 흐름의 힘입니다.

과거 세션이 다른 DB에 있으면,
같은 시스템처럼 보여도 예전 맥락은 약해집니다.
그러면 사용자는 이렇게 느낍니다.
- 왜 저번 얘기를 못 찾지?
- 왜 예전 흐름이 안 이어지지?

하지만 운영자는 이렇게 번역해야 합니다.
- 지금 어떤 profile인가?
- 어느 state.db를 보고 있나?
- session 회수 경로가 달라진 건 아닌가?

### 2. profile 경계가 바뀌었을 수 있다
profile은 단순 이름표가 아닙니다.
보통 아래를 함께 가릅니다.
- config
- state.db
- session history
- installed skills
- SOUL / AGENTS
- memory context
- gateway 실행 상태

즉 profile이 바뀌면,
사용자는 "하비가 다르게 느껴진다"고 말하고,
운영자는 "다른 저장소와 규칙 묶음을 보고 있구나"라고 봐야 합니다.

### 3. skill 구성 차이일 수도 있다
전에 되던 작업이 지금은 자꾸 낯설게 느껴진다면,
그건 session이 약한 게 아니라 **skill 부재**일 수 있습니다.

해결 절차가 재사용 가능한 형태로 안 남아 있으면,
매번 비슷한 일을 새로 푸는 느낌이 생깁니다.
그 체감은 쉽게 "기억이 약해졌다"로 번역됩니다.

![체감 변화는 저장소와 경계 문제로 번역된다](/assets/build-journey/why-same-harvey-feels-like-different-memory/article-figure-02-boundaries-and-storage.png)

## memory에는 다 넣으면 되는 걸까
여기서 또 자주 하는 오해가 있습니다.
기억이 부족해 보이면 memory를 계속 늘리면 된다고 생각하는 겁니다.

하지만 memory는 많이 넣는다고 좋아지지 않습니다.
오히려 반대입니다.
**오래 가는 선호와 규칙만 짧게 남길수록** 잘 작동합니다.

memory에 넣으면 안 되는 대표적인 것:
- 작업 로그
- 긴 사건 설명
- 일회성 TODO
- 해결 절차

왜냐하면 이런 건 각각 더 맞는 레이어가 있기 때문입니다.
- 과거 대화 = session
- 절차 = skill
- 긴 설명 = Topic / 위키 문서

즉 memory의 적은 부족함보다 **과잉 적재**입니다.

---

## 그래서 어디부터 점검해야 하나
이상 체감이 생기면 이 순서가 좋습니다.

### 1. 지금 어떤 profile이 활성인가
- 지금 누구로 실행 중인가?
- gateway는 어느 profile 기준으로 돌고 있나?

### 2. 어느 state.db를 보고 있는가
- 과거 세션 회수 경로는 어디인가?
- 예전 맥락이 다른 DB에 남아 있지는 않은가?

### 3. skill 구성이 같은가
- 전에 되던 작업 방식이 지금도 같은가?
- 재사용 절차가 빠져 있지는 않은가?

### 4. memory에 뭘 넣고 뭘 안 넣었는가
- 이건 장기 선호인가?
- 작업 로그를 memory로 착각하고 있지는 않은가?

즉,
"기억이 이상하다"는 감각이 들면
먼저 성능을 의심하기보다 **레이어와 경계부터 확인하는 습관**이 필요합니다.

![먼저 무엇을 봐야 하는가](/assets/build-journey/why-same-harvey-feels-like-different-memory/article-figure-03-diagnostic-order.png)

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 과거 대화는 session으로 찾는다
session은 과거 회수 층입니다.
예전 대화를 다시 찾고 싶을 때는 session을 봐야 합니다.

### 원칙 2. 오래 갈 선호는 memory에 남긴다
호칭, 스타일, 반복 규칙처럼 다음에도 계속 유효한 것만 memory 후보입니다.

### 원칙 3. 반복 작업 절차는 skill로 만든다
잘 해결한 방법을 memory에 넣는 대신,
다음에도 쓸 workflow라면 skill로 만드는 게 맞습니다.

### 원칙 4. profile 전환은 가볍게 보지 않는다
profile이 달라지면 단순 말투 변화가 아니라,
저장소와 규칙 세트가 달라질 수 있습니다.

## 결론
같은 하비인데 기억이 다르게 느껴질 때,
우리가 가장 먼저 놓치기 쉬운 건 이것입니다.

**기억은 하나가 아니라 층이라는 사실.**

session은 과거 대화,
memory는 장기 선호,
skill은 작업 절차,
profile과 state.db는 그 기억이 어디서 어떻게 작동하는지를 결정하는 경계입니다.

이걸 한 통으로 보면 하비가 이상해 보이고,
분리해서 보면 운영이 풀립니다.

그래서 중요한 건 기억을 더 많이 넣는 게 아니라,
**기억의 레이어를 정확히 나눠 보는 것**입니다.

## FAQ
### 1. 전에 얘기한 내용은 다 memory에 저장되나요?
아닙니다. 단순 대화 내용은 session에만 남아 있을 수 있습니다. memory는 장기 선호와 규칙 중심입니다.

### 2. 해결한 방법은 memory에 넣어야 하나요?
대부분은 skill이 더 맞습니다. 해결 절차는 재사용 가능한 workflow로 남기는 편이 좋습니다.

### 3. 같은 하비인데 왜 예전 대화를 못 찾을 수 있나요?
다른 profile이나 다른 state.db를 보고 있을 가능성이 큽니다. 체감 문제보다 저장소 경계를 먼저 봐야 합니다.

### 4. memory가 가득 차면 어떻게 해야 하나요?
더 넣기 전에 정리해야 합니다. 장기 선호가 아닌 것, 작업 로그, 긴 사건 설명은 다른 레이어로 보내는 게 맞습니다.

### 5. 기억이 이상해 보일 때 가장 먼저 봐야 할 건 뭔가요?
profile, state.db, session 회수 경로, skill 구성 같은 경계부터 점검하는 게 좋습니다.
