# Google Workspace 연동은 왜 늘 생각보다 오래 걸릴까

## TL;DR
Google Workspace를 에이전트에 붙이는 일은 겉으로 보면 단순해 보입니다. Gmail 읽고, Calendar 보고, Docs와 Drive를 연결하면 될 것 같으니까요. 그런데 실제로는 늘 생각보다 오래 걸립니다. 이유는 기능이 부족해서가 아니라, **인증 방식, 공식성, 토큰 저장, MCP 입출력 계약, on-demand 운영 모델** 같은 운영 조건이 먼저 맞아야 하기 때문입니다. 결국 이 문제는 API 연결보다 **운영 모델 선택**에서 더 자주 터집니다.

## 이 글의 답
답부터 말하면 이렇습니다.

Google Workspace 연동이 늘 생각보다 오래 걸리는 이유는,
우리가 기능을 붙이는 일이라고 생각하지만 실제로는 **인증과 운영 모델을 설계하는 일**에 더 가깝기 때문입니다.

핵심은 네 가지입니다.
- 기능 수보다 운영 적합성이 더 중요하다.
- 공식성/신뢰도 기준이 생각보다 큰 변수다.
- MCP는 만능 연결층이 아니라 입출력 계약이 깨지면 가장 먼저 흔들린다.
- 초기에는 상시 자동화보다 on-demand 운영이 더 안전할 때가 많다.

즉,
이건 “어떤 도구가 더 세냐”보다 **어떤 방식이 지금 운영에 덜 위험하냐**를 고르는 문제입니다.

## 사람들이 처음에 가장 많이 단순하게 보는 지점
처음엔 다 비슷하게 생각합니다.

- Gmail 읽고 쓰면 되지
- Calendar 이벤트 넣으면 되지
- Docs / Drive 연결하면 되지

겉으로만 보면 맞습니다.
하지만 실제 운영에서는 이 질문들이 먼저 옵니다.

- 누가 인증하는가
- 어디에 토큰이 저장되는가
- 브라우저 로그인 흐름을 허용할 것인가
- 상시 자동화인가, 필요할 때만 호출할 것인가
- 실패했을 때 어디부터 디버깅할 것인가

즉,
문제는 기능이 아니라 **운영 가정**입니다.

---

## 왜 이 문제는 기능보다 운영 모델에서 더 자주 꼬이나
### 1. 기능은 비슷해 보여도 인증 부담은 다르다
Google Cloud + OAuth 직접 구성,
gws CLI 계열,
Gemini Workspace / 공식성 높은 경로는
겉으로 보면 다 비슷한 연결 수단처럼 보일 수 있습니다.

하지만 실제 차이는 기능 수보다 아래에서 납니다.
- 공식성/신뢰도
- 초기 설치 난이도
- OAuth/토큰 경험
- on-demand 적합성
- 상시 자동화 확장성
- 장애 시 디버깅 용이성

즉,
도구 비교표만 보고 고르면 답이 안 나옵니다.
연동 방식은 결국 **운영 철학 선택**입니다.

### 2. 실무에선 “제일 강한 방식”보다 “제일 덜 위험한 방식”이 먼저다
직접 구성은 강력합니다.
하지만 강력하다는 건,
그만큼 세팅 부담과 설명 부담도 크다는 뜻입니다.

빠른 CLI 실험은 매력적입니다.
하지만 공식성 신호가 약하고 환경 의존성이 크면,
실사용 단계에서 다시 불안해집니다.

그래서 실제 운영에서는 종종 이렇게 정리됩니다.
- 제어권이 높다고 바로 좋은 건 아니다.
- 빨리 붙는다고 장기 운영에 맞는 것도 아니다.
- 결국 지금 내 운영에 **덜 위험한 경로**를 먼저 택하는 쪽이 낫다.

![기능보다 운영 모델이 먼저다](/assets/build-journey/why-google-workspace-integration-takes-longer-than-expected/article-figure-01-ops-model-first.png)

## 왜 MCP는 붙었는데 실제로는 잘 깨질까
여기서 두 번째 함정이 나옵니다.
많은 사람이 MCP를 붙이면 연결이 매끄러워질 거라고 기대합니다.

하지만 MCP는 만능 연결층이 아닙니다.
오히려 **정해진 형식의 입출력 계약**에 예민한 계층입니다.

예를 들어 이런 일이 생깁니다.
- OAuth 안내문이 stdout에 섞임
- 브라우저 로그인 메시지가 출력됨
- keychain이 안 돼서 file fallback 발생
- 도구는 살아 있는데 파싱이 깨짐

사용자는 "도구가 안 된다"고 느끼지만,
실제로는 도구가 죽은 게 아니라 **계약이 깨진 것**일 수 있습니다.

즉,
MCP 장애는 종종 기능 실패가 아니라
**프로토콜 오염** 문제입니다.

![MCP는 도구보다 계약이 먼저다](/assets/build-journey/why-google-workspace-integration-takes-longer-than-expected/article-figure-02-contract-breaks-first.png)

## 왜 초반엔 on-demand가 더 낫나
실무에선 여기서 한 번 더 갈립니다.
상시 자동화를 바로 붙일 것인가,
필요할 때만 호출하게 만들 것인가.

우리 경험상 초반엔 on-demand가 더 나은 경우가 많았습니다.

이유는 단순합니다.
- 인증 부담을 더 보수적으로 다룰 수 있다.
- 실패 시 원인을 좁혀 보기 쉽다.
- 사용자가 실제로 자주 쓰는 흐름부터 검증할 수 있다.
- 자동화 범위를 너무 빨리 넓혀 생기는 사고를 줄일 수 있다.

즉,
Google Workspace 연동은 처음부터 “전부 자동화”보다
**자주 쓰는 일부터, 필요할 때 정확히 호출**하는 쪽이 더 안정적입니다.

---

## 그럼 실제로는 무엇부터 봐야 하나
이상하게 오래 걸리거나 자꾸 깨지면,
보통 이 순서로 보면 됩니다.

### 1. 지금 필요한 건 상시 자동화인가 on-demand인가
먼저 운영 모델부터 정합니다.
이게 안 정해지면 이후 판단이 다 흔들립니다.

### 2. 공식성 기준을 먼저 정한다
공식/대중적 auth를 우선할지,
실험적이지만 빠른 경로를 감수할지 먼저 정해야 합니다.

### 3. 토큰과 인증 저장 방식을 본다
- 로컬 토큰 보관 허용 여부
- keychain 의존 여부
- fallback 허용 여부

여기서부터 환경 차이가 크게 납니다.

### 4. 실패 시 어느 계층을 볼지 정한다
- auth 문제인가
- MCP 포맷 문제인가
- CLI/runtime 문제인가

이걸 구분하지 않으면 모든 게 "연동이 안 됨"으로 뭉개집니다.

![실전 점검 순서](/assets/build-journey/why-google-workspace-integration-takes-longer-than-expected/article-figure-03-check-order.png)

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 기능보다 운영 모델을 먼저 본다
이게 제일 중요합니다.
상시 자동화인지 on-demand인지 먼저 정해야 합니다.

### 원칙 2. 공식성 높은 경로를 우선 검토한다
처음엔 특히 그렇습니다.
설명 가능성과 신뢰도가 더 중요합니다.

### 원칙 3. MCP 오류는 계약 실패일 수 있다고 본다
도구가 죽은 것처럼 보여도,
실제로는 stdout 오염이나 출력 포맷 문제일 수 있습니다.

### 원칙 4. 초기에는 on-demand가 이긴다
최소한 초반 안정화 구간에서는 그렇습니다.
너무 빨리 자동화를 넓히면 오히려 운영이 흔들립니다.

## 결론
Google Workspace 연동이 늘 생각보다 오래 걸리는 이유는,
우리가 계속 기능 연결 문제로 보기 때문입니다.

하지만 실제 핵심은 이겁니다.
**Google 연동의 본질은 API 연결이 아니라 인증과 운영 모델 선택이다.**

그래서 실무에서는
제일 강한 방식보다,
**지금 운영에 가장 덜 위험한 방식**을 먼저 고르는 쪽이 더 낫습니다.

그리고 MCP가 흔들릴 때도,
도구가 죽었다고 단정하기보다
**계약이 먼저 깨졌을 수 있다**고 보는 습관이 중요합니다.

## FAQ
### 1. 왜 공식성이 중요한가요?
공식성은 단순 취향 문제가 아니라, 설명 가능성과 장기 운영 신뢰도에 영향을 줍니다.

### 2. 왜 MCP는 붙었는데 파싱 오류가 날 수 있나요?
인증 안내문, 로그인 메시지, stdout 오염이 끼면 MCP가 기대하는 형식 계약이 깨질 수 있기 때문입니다.

### 3. 왜 keychain fallback이 생기나요?
환경에 따라 keychain 접근이 매끄럽지 않으면 file 기반 fallback이 생길 수 있습니다. 이건 단순 오류가 아니라 보안/편의 균형을 다시 판단해야 하는 신호이기도 합니다.

### 4. 왜 처음엔 on-demand가 더 낫나요?
인증 부담과 실패 원인을 더 좁혀 볼 수 있고, 실제로 자주 쓰는 흐름부터 검증할 수 있기 때문입니다.

### 5. 실전에서 가장 먼저 봐야 할 것은 무엇인가요?
상시 자동화인지 on-demand인지, 즉 운영 모델부터 먼저 정하는 게 제일 중요합니다.
