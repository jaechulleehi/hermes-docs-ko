# 에르메스는 단순 챗봇이 아니라 운영 시스템이다

## TL;DR
겉으로 보면 Hermes도 그냥 대화를 잘하는 AI처럼 보일 수 있습니다. 질문을 받고 답하고, 도구를 쓰고, 메모를 남기니까요. 그런데 실제로는 이걸 챗봇이라고만 보면 중요한 걸 놓칩니다. Hermes의 핵심은 답변 한 번의 품질보다 **요청을 어떻게 해석하고, 누가 조사하고, 누가 정리하고, 어떤 프로필과 기억을 쓰고, 무엇을 검증하고, 무엇을 기록으로 남길지**를 하나의 구조로 묶었다는 데 있습니다. 그래서 Hermes는 단순 챗봇이라기보다 **운영 시스템**에 더 가깝습니다.

## 이 글의 답
답부터 말하면 이렇습니다.

Hermes를 운영 시스템으로 봐야 하는 이유는,
이 시스템이 단순히 말 잘하는 AI가 아니라 **창구, 역할, 상태, 기억, 도구, 기록을 함께 굴리는 구조**이기 때문입니다.

핵심은 다섯 가지입니다.
- 메인 창구가 요청 해석과 라우팅을 책임진다.
- 역할 분리형 구조가 조사·정리·최종 판단을 나눈다.
- 프로필은 캐릭터가 아니라 상태 저장 단위다.
- 기억은 session / memory / skill / identity로 레이어가 나뉜다.
- 도구 실행과 기록 자산화까지 연결되어야 실제 운영이 된다.

즉,
이건 똑똑한 답변 하나의 문제가 아니라 **지속적으로 굴러가는 운영 구조를 어떻게 만들 것인가**의 문제입니다.


---

## 왜 챗봇이라고만 보면 자꾸 설명이 어긋날까
많은 사람이 AI 시스템을 볼 때 먼저 이렇게 생각합니다.
- 질문하면 답한다
- 잘 말하면 좋은 AI다
- 기능이 많으면 더 좋은 시스템이다

이 관점 자체가 완전히 틀린 건 아닙니다.
하지만 Hermes를 여기에만 맞춰 보면 자꾸 설명이 어긋납니다.

예를 들어 실제로 중요한 건 이런 것들입니다.
- 누가 이 요청을 먼저 받는가
- 조사와 정리를 같은 방식으로 처리해도 되는가
- 어떤 profile이 살아 있는가
- 지금의 답이 session 때문인가 memory 때문인가 skill 때문인가
- 실제로 확인하고 실행했는가
- 이 결과를 나중에 다시 찾을 수 있게 남겼는가

즉,
Hermes의 핵심은 한 번의 대화가 아니라
**대화를 포함한 운영 전체 흐름**입니다.

---

## 메인 창구가 왜 이 시스템의 시작점인가
여기서 첫 번째 핵심이 나옵니다.
Hermes는 여러 역할이 있어도 사용자 앞에서는 하비 같은 메인 창구가 먼저 받습니다.

이게 중요한 이유는 단순합니다.
- 요청 해석을 한 곳에서 한다.
- 누가 조사할지, 누가 정리할지 사용자가 매번 고르지 않아도 된다.
- 결과가 흩어지지 않고 한 창구로 모인다.

즉,
좋은 운영 시스템은 기능이 많아 보이는 것보다
**입구가 단순한 것**이 더 중요할 때가 많습니다.

그래서 Hermes를 챗봇보다 운영 시스템이라고 보는 순간,
하비는 캐릭터가 아니라 **front door이자 orchestrator**로 이해됩니다.


---

## 왜 역할 분리와 프로필 경계가 같이 중요할까
두 번째 핵심은 역할과 상태를 한 덩어리로 보지 않는다는 점입니다.

Hermes에선 보통 이런 구분이 들어갑니다.
- 조사형
- 정리형
- 총괄형

그리고 profile에 따라 달라질 수 있는 것도 분명합니다.
- config
- state DB
- session history
- installed skills
- identity docs
- gateway 상태

이 말은,
Hermes가 단순히 여러 봇이 있는 시스템이라는 뜻이 아닙니다.
**역할 분리와 상태 경계를 운영 단위로 본다**는 뜻입니다.

그래서 같은 모델을 써도 다르게 느껴질 수 있고,
같은 하비인데 기억이 다르게 느껴지는 일도 구조적으로 설명할 수 있습니다.

즉,
좋은 운영 시스템은 개성을 늘리는 게 아니라
**경계를 선명하게 만드는 방식**으로 안정성을 얻습니다.

---

## 기억을 한 통으로 다루지 않는 이유
세 번째 핵심은 기억의 레이어입니다.
Hermes는 기억을 마법처럼 한 덩어리로 보지 않습니다.

보통 이렇게 나눕니다.
- session = 과거 대화/작업 로그
- memory = 장기 선호와 규칙
- skill = 재사용 절차
- identity docs = 역할과 원칙

이게 왜 중요하냐면,
문제가 생겼을 때 어디를 봐야 하는지가 달라지기 때문입니다.
- 지금 이 말투는 user preference 때문인가?
- 이 행동은 특정 skill 때문인가?
- 이 맥락은 session history 때문인가?
- 이 방향성은 identity docs 때문인가?

즉,
Hermes는 기억을 똑똑해 보이게 포장하기보다
**운영 가능한 층으로 분해**해서 다룹니다.


---

## 도구 실행과 기록 자산화까지 붙어야 왜 비로소 운영이 되나
네 번째 핵심은 실행과 기록입니다.
좋은 답을 말하는 것과,
실제로 확인하고 고치고 남기는 것은 다릅니다.

Hermes가 운영 시스템에 가까운 이유는,
대체로 여기까지 묶기 때문입니다.
- 파일 읽기/수정
- 프로세스 확인
- 브라우저 확인
- 일정/도구 연동
- 결과 검증
- Obsidian 기록
- lesson learned와 블로그/강의 자산화

즉,
대화를 잘하는 AI에서 끝나는 게 아니라
**실행 → 검증 → 기록 → 재사용**으로 이어져야 운영 시스템이 됩니다.

이게 빠지면 결국 대화는 휘발되고,
시스템은 반복 학습 대신 반복 삽질을 하게 됩니다.

---

## 그래서 Hermes를 어떻게 설명하는 게 맞나
우리 기준에선 이렇게 설명하는 편이 가장 정확합니다.

### 1. 챗봇이 아니라 메인 창구가 있는 운영 시스템
하비는 단순 페르소나가 아니라 요청의 입구입니다.

### 2. 멀티봇이 아니라 역할 분리형 구조
조사, 정리, 최종 판단을 같은 근육으로 처리하지 않습니다.

### 3. 캐릭터가 아니라 상태 경계를 다루는 체계
profile, state DB, history, skill이 운영 단위로 나뉩니다.

### 4. 기억을 마법이 아니라 레이어로 보는 체계
session / memory / skill / docs를 분리해서 봅니다.

### 5. 답변이 아니라 실행과 기록까지 포함하는 체계
도구 검증과 Obsidian 자산화까지 연결합니다.

핵심은,
Hermes를 챗봇으로만 설명하면 구조가 사라지고,
운영 시스템으로 설명해야 왜 이런 설계를 했는지가 보인다는 점입니다.


---

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 메인 창구를 먼저 세운다
사용자가 에이전트 운용법부터 고민하게 만들지 않습니다.

### 원칙 2. 역할 분리와 상태 경계를 같이 본다
누가 무엇을 하는지와, 어떤 상태를 보는지를 분리합니다.

### 원칙 3. 기억은 레이어로 다룬다
session, memory, skill, docs를 섞어 설명하지 않습니다.

### 원칙 4. 도구로 확인하고 실행한다
설명보다 실제 검증을 우선합니다.

### 원칙 5. 결과를 기록 자산으로 남긴다
대화를 휘발시키지 않고 위키/블로그/강의로 연결합니다.

---

## 결론
Hermes를 단순 챗봇이라고 부르면,
이 시스템의 중요한 절반이 빠집니다.

핵심은 이겁니다.
**Hermes는 질문에 답하는 AI가 아니라, 요청을 해석하고, 역할을 태우고, 상태를 관리하고, 도구로 검증하고, 기록으로 남기는 운영 시스템이다.**

그래서 Hermes의 경쟁력은 단순히 답변이 그럴듯한가에만 있지 않습니다.
**운영이 계속 굴러가고, 다음에도 다시 붙을 수 있는 구조를 만들었는가**에 있습니다.

## FAQ
### 1. 챗봇과 운영 시스템의 차이는 뭔가요?
챗봇은 대화 자체에 초점이 있지만, 운영 시스템은 창구, 라우팅, 상태, 실행, 기록까지 함께 다룹니다.

### 2. 왜 메인 창구가 그렇게 중요한가요?
요청 해석과 결과 통합을 한 곳에서 해야 사용자가 덜 헤매고 품질도 덜 흩어지기 때문입니다.

### 3. 왜 profile을 캐릭터가 아니라 상태 단위로 보나요?
config, state DB, session history, installed skills가 실제로 달라질 수 있기 때문입니다.

### 4. 왜 기억을 한 통으로 다루지 않나요?
문제가 생겼을 때 원인을 더 정확히 찾고, 운영 기준을 더 선명하게 설명하기 위해서입니다.

### 5. 왜 Obsidian 기록까지 붙어야 하나요?
실행과 시행착오가 기록과 자산화로 이어져야 시스템이 다음에도 더 똑똑하게 굴러가기 때문입니다.
