# 에르메스에서 스킬은 언제 만들고 어떻게 관리해야 할까

## TL;DR
스킬은 좋아 보여서 만드는 문서가 아닙니다. Hermes에서 스킬은 procedural memory, 즉 **다음에도 다시 써야 하는 절차를 덜 헤매게 만드는 재사용 단위**에 가깝습니다. 그래서 너무 이르게 만들면 오히려 관리 비용이 커지고, 만들어 놓고 방치하면 부채가 됩니다. 핵심은 **재사용 가치가 생겼을 때 만들고, 실제 사용 중 바로 패치하며, 겹치면 통합하고, 오래되면 폐기하는 것**입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

Hermes에서 스킬은,
복잡한 작업을 다시 풀지 않게 해주는 재사용 가치가 생겼을 때 만들고,
만든 뒤에는 최신성·범위·연결성 기준으로 계속 관리해야 합니다.

핵심은 다섯 가지입니다.
- 스킬 생성 기준은 멋짐이 아니라 재사용 가치다.
- 5단계 이상 복잡한 작업, 시행착오 끝에 안정된 절차가 생겼을 때 만들기 좋다.
- 너무 일회성인 사건이나 감각적 팁은 스킬화하지 않는 편이 낫다.
- 만든 스킬은 패치, 통합, 폐기까지 포함해 유지해야 한다.
- wiki는 설명, skill은 절차라는 경계를 지켜야 자산이 된다.

즉,
좋은 스킬 운영은 많이 만드는 일이 아니라 **필요할 때 만들고, 맞게 유지하는 일**입니다.

## 왜 스킬은 많을수록 좋은 게 아닐까
처음엔 스킬이 많아지면 좋아 보입니다.
한 번 만든 절차를 재사용할 수 있으니까요.

하지만 실제 운영에선 이런 일이 쉽게 생깁니다.
- 아직 검증 안 된 감각적 팁을 스킬로 만든다.
- 비슷한 스킬이 여러 개 생긴다.
- 경로와 명령이 바뀌었는데 그대로 남아 있다.
- wiki에 있어야 할 설명까지 스킬에 밀어 넣는다.

그러면 스킬은 자산이 아니라,
오히려 **잘못된 자동화의 출발점**이 됩니다.

즉,
스킬은 많이 만드는 게 중요한 게 아니라
**언제 만들어야 하는지와 만든 뒤 어떻게 유지할지**가 더 중요합니다.

---

## 언제 만들어야 하나: 재사용 가치가 생겼을 때
여기서 첫 번째 핵심이 나옵니다.
Hermes에서 스킬은 보통 이런 순간에 만드는 편이 좋습니다.

- 5단계 이상 되는 복잡한 작업을 해결했을 때
- 시행착오 끝에 안정된 절차가 생겼을 때
- 다음에도 비슷한 작업이 반복될 가능성이 높을 때
- 도구/환경 의존성이 있어 다시 헤맬 가능성이 클 때
- 사용자나 팀이 다시 같은 걸 물을 가능성이 클 때

반대로 아직 만들면 안 되는 경우도 분명합니다.
- 너무 일회성인 사건
- 아직 검증되지 않은 감각적 팁
- 작업이 너무 짧고 단순할 때
- 사용자 선호를 procedural memory로 착각할 때

즉,
좋은 스킬은 "멋져 보여서"가 아니라
**다음에도 다시 써야 하고 다시 안 헤매고 싶을 때** 만들어집니다.

![스킬은 언제 만들어야 하나](/assets/build-journey/when-and-how-to-manage-skills-in-hermes/article-figure-01-when-to-create.png)

## 만든 뒤가 더 중요하다: 패치하지 않으면 부채가 된다
두 번째 핵심은 생성보다 관리입니다.
스킬은 한 번 만들었다고 끝나지 않습니다.

실제로 중요한 건 이쪽입니다.
- 경로가 여전히 맞는가
- 명령이 여전히 통하는가
- 운영 원칙이 바뀌지 않았는가
- 언제 써야 하고 언제 쓰면 안 되는가가 선명한가
- 관련 wiki 문서와 겹치지 않는가

이걸 놓치면,
성공 직후엔 좋아 보였던 스킬이 시간이 지나면서 부채가 됩니다.

그래서 Hermes에서 좋은 스킬 관리는 문서 정리가 아니라
**운영 기준 유지보수**에 가깝습니다.

---

## 그럼 어떻게 관리해야 하나
우리 기준에선 보통 이 루프로 보는 편이 제일 실용적입니다.

### 1. Create
재사용 가치가 생긴 절차만 스킬로 만든다.

### 2. Use
실제 작업에 다시 써본다.

### 3. Patch
문제, 누락, 오래된 명령이 보이면 바로 수정한다.

### 4. Merge
겹치는 스킬이 생기면 통합을 검토한다.

### 5. Archive / Remove
더 이상 안 쓰거나 구조가 바뀌면 정리한다.

핵심은,
스킬을 박제된 문서로 두지 않고
**사용 중 계속 살아 있는 절차 기억**으로 다루는 겁니다.

![스킬 유지보수 루프](/assets/build-journey/when-and-how-to-manage-skills-in-hermes/article-figure-02-skill-maintenance-loop.png)

---

## 어떤 순간에 스킬이 부채가 되나
세 번째 핵심은 부채 신호를 빨리 보는 겁니다.
이런 신호가 보이면 관리가 필요합니다.

- 같은 일을 설명하는 스킬이 여러 개 있다
- 실제 환경과 안 맞는 경로가 남아 있다
- wiki와 skill 경계가 섞여 있다
- "이거 지금도 맞나?"가 반복된다
- 실제 사용 중 자꾸 예외 처리만 늘어난다

이런 상태는 procedural memory가 강해진 게 아니라,
오히려 운영 구조가 흐려지고 있다는 신호입니다.

즉,
좋은 스킬 운영은 추가보다
**정리와 패치의 감각**이 더 중요할 때가 많습니다.

![스킬이 부채가 되는 신호](/assets/build-journey/when-and-how-to-manage-skills-in-hermes/article-figure-03-skill-debt-signals.png)

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 재사용 가치가 생길 때 만든다
반복 가능성과 검증 가능성이 생긴 뒤에 만든다.

### 원칙 2. 해결 직후가 가장 좋다
시행착오와 주의점이 아직 선명할 때 남긴다.

### 원칙 3. wiki와 skill 경계를 지킨다
wiki는 설명과 기준, skill은 절차와 검증을 맡긴다.

### 원칙 4. 실제 사용 중 바로 패치한다
문제를 발견했는데 나중으로 미루지 않는다.

### 원칙 5. 겹치면 통합하고 오래되면 정리한다
스킬 수를 늘리는 것보다 구조를 선명하게 유지한다.

---

## 결론
Hermes에서 스킬은,
많이 모아두는 장식이 아닙니다.

핵심은 이겁니다.
**스킬은 다시 써야 하는 절차를 다시 안 헤매게 만드는 procedural memory다.**

그래서 중요한 건 많이 만드는 게 아니라,
언제 만들어야 하는지 알고,
만든 뒤에는 실제 사용 기준으로 계속 맞게 유지하는 것입니다.

그렇게 해야 스킬은 자산이 되고,
그렇지 않으면 금방 부채가 됩니다.

## FAQ
### 1. 스킬은 성공한 작업마다 다 만들어야 하나요?
아닙니다. 재사용 가치와 반복 가능성이 생긴 작업만 스킬로 만드는 편이 좋습니다.

### 2. 왜 일회성 작업은 스킬화하면 안 되나요?
관리 비용만 늘고, 실제로는 다시 쓰지 않을 가능성이 크기 때문입니다.

### 3. wiki와 skill의 차이는 뭔가요?
wiki는 설명과 기준, skill은 실제 절차와 검증 단위를 더 강하게 맡습니다.

### 4. 오래된 스킬은 어떻게 해야 하나요?
패치, 통합, archive, remove 중 무엇이 맞는지 판단해 정리해야 합니다.

### 5. 가장 먼저 바꿔야 할 습관은 무엇인가요?
좋은 해결을 했을 때 "이걸 다음에도 다시 쓸까?"를 먼저 묻는 습관입니다.
