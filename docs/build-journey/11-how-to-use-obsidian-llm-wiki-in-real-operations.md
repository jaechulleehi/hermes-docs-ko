# Obsidian LLM Wiki를 실제 운영에 어떻게 활용할 것인가

## TL;DR
Obsidian LLM Wiki는 긴 문서를 저장하는 예쁜 노트 앱 운용법이 아닙니다. 실제 운영에선 **반복 질문, 운영 기준, 사례, FAQ 후보, 파생 문서**를 연결해 사람이든 에이전트든 다시 찾고 다시 쓰기 쉬운 지식베이스로 굴려야 가치가 생깁니다. 핵심은 많이 쌓는 게 아니라 **질문 중심으로 연결되게 쌓고, memory와 역할을 나누고, 블로그·강의·FAQ까지 파생 가능한 구조를 만드는 것**입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

Obsidian LLM Wiki를 실제 운영에서 잘 쓰는 방법은,
문서를 많이 만드는 게 아니라 **반복 질문과 운영 기준이 다시 연결되게 쌓이는 구조**로 운영하는 것입니다.

핵심은 다섯 가지입니다.
- memory는 짧고 오래 가는 규칙을, wiki는 길고 연결되는 지식을 맡긴다.
- wiki는 질문 중심으로 써야 나중에 다시 찾기 쉽다.
- 사례와 기준, 연결 포인트와 파생 후보를 같이 남겨야 한다.
- 단일 노트보다 문서군과 연결망으로 운영해야 한다.
- 실제 질문이 들어왔을 때 wiki에서 바로 블로그/강의/FAQ로 다시 뽑을 수 있어야 한다.

즉,
LLM Wiki의 본질은 저장이 아니라 **재탐색과 재사용이 가능한 운영 지식 구조**입니다.


---

## 왜 문서를 많이 쌓아도 다시 못 쓰는 일이 생길까
처음엔 다들 비슷하게 생각합니다.
- 일단 적어두자
- 나중에 찾으면 되겠지
- 노트가 많아지면 지식도 쌓이겠지

그런데 실제 운영에선 자주 이렇게 됩니다.
- 그거 어디 있었지?
- 비슷한 문서가 여러 개라 기준점이 안 보인다.
- memory에 있었는지 wiki에 있었는지 헷갈린다.
- 문서는 있는데 블로그나 FAQ로 다시 쓰기가 어렵다.

즉,
문서 수가 늘어난다고 곧바로 지식 시스템이 되는 건 아닙니다.
LLM Wiki의 핵심은 양이 아니라 **다시 붙는 구조**입니다.

---

## memory와 wiki를 왜 나눠서 봐야 하나
여기서 첫 번째 핵심이 나옵니다.
Hermes 운영에선 memory와 wiki를 같은 걸로 보면 금방 꼬입니다.

- memory = 짧고 오래 가는 선호/규칙
- wiki = 길고 연결되는 설명/사례/기준

긴 사건 설명이나 복합 비교 문서를 memory에 넣기보다,
wiki에 남기고 memory에는 핵심 원칙만 남기는 편이 훨씬 낫습니다.

이렇게 나누면 좋은 점이 분명합니다.
- memory는 가볍고 오래 간다
- wiki는 깊고 연결된다
- 둘의 역할이 겹치지 않는다

즉,
좋은 LLM Wiki 운영은 문서 전략이기도 하지만,
동시에 **기억 전략 분리**이기도 합니다.


---

## 어떤 문서가 실제로 살아남는 wiki가 되나
두 번째 핵심은 질문 중심입니다.
좋은 LLM Wiki 문서는 보통 이렇게 생깁니다.
- 이 문서는 무엇에 답하는가
- 앞부분에 먼저 답이 있는가
- 사례와 기준이 같이 있는가
- 다른 문서로 연결되는가
- 블로그/강의/FAQ로 뽑아낼 후보가 있는가

반대로 나쁜 패턴도 분명합니다.
- 그냥 노트만 많이 만든다
- 연결 없이 쌓는다
- 질문 없는 사건 기록만 남긴다
- 파생 없이 원천 문서만 쌓고 활용하지 않는다

즉,
LLM Wiki는 노트 모음이 아니라
**질문과 연결을 기준으로 살아남는 문서군**입니다.

---

## 실제 운영에선 어떻게 활용해야 하나
우리 기준에선 보통 이런 흐름으로 씁니다.

### 1. 새로운 질문이 오면 wiki에서 관련 주제를 찾는다
memory만 보지 않고, 주제 문서를 같이 본다.

### 2. 비슷한 시행착오가 나오면 기존 문서에 붙인다
새 노트를 남발하기보다 기존 축을 강화한다.

### 3. 문서 안에 파생 후보를 남긴다
블로그 제목 후보, FAQ 후보, 강의 포인트를 같이 적어둔다.

### 4. 위키에서 바로 블로그와 강의를 뽑는다
원천 문서를 source note로 써서 재가공 비용을 줄인다.

### 5. 축으로 묶어 운영한다
전환사, 역할 구조, 기억/프로필, 콘텐츠 파이프라인 같은 문서군으로 본다.

핵심은,
wiki를 저장소로 방치하지 않고
**질문이 오면 다시 붙고, 새 사례가 나오면 다시 강화되는 운영 루프**로 돌리는 겁니다.


---

## 왜 Obsidian이 실제 운영에서 잘 맞나
세 번째 핵심은 연결성입니다.
Obsidian이 좋은 이유는 예쁘게 적을 수 있어서가 아니라,
문서를 연결하고 문서군을 운영하기에 적합하기 때문입니다.

실전에서 특히 중요한 건 이 부분입니다.
- Inbox → Topic → Program → Session → Distribution 구조
- 문서 간 wikilink
- source note를 중심으로 한 파생 설계
- 반복 질문을 기존 축에 붙이는 운영 방식

즉,
Obsidian LLM Wiki는 단순 노트 앱 활용이 아니라
**운영 지식의 source of truth를 만드는 방식**에 가깝습니다.

그래서 블로그와 강의가 따로 노는 게 아니라,
같은 주제 문서에서 계속 파생될 수 있습니다.


---

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 질문 중심으로 쓴다
문서가 무엇에 답하는지 선명해야 다시 찾기 쉽다.

### 원칙 2. 먼저 답을 둔다
긴 문서라도 초반에 핵심이 보여야 파생이 쉬워진다.

### 원칙 3. 사례와 기준을 같이 둔다
설명만 있거나 사례만 있으면 재사용이 약해진다.

### 원칙 4. 연결 포인트와 파생 후보를 넣는다
문서가 혼자 있지 않게 만든다.

### 원칙 5. memory와 wiki의 역할을 섞지 않는다
짧은 규칙과 긴 설명을 다른 층에서 관리한다.

---

## 결론
Obsidian LLM Wiki를 실제 운영에 활용한다는 건,
긴 문서를 저장하는 습관을 말하는 게 아닙니다.

핵심은 이겁니다.
**질문, 기준, 사례, 파생 경로가 다시 연결되게 쌓이는 운영 지식 구조를 만드는 것**입니다.

그래서 좋은 LLM Wiki는 노트가 많은 시스템이 아니라,
다음 질문이 왔을 때 바로 다시 붙고,
그 문서에서 블로그와 강의까지 다시 뽑아낼 수 있는 시스템입니다.

## FAQ
### 1. LLM Wiki는 memory와 뭐가 다른가요?
memory는 짧고 오래 가는 규칙, wiki는 길고 연결되는 설명과 사례를 더 잘 맡습니다.

### 2. 왜 질문 중심으로 써야 하나요?
그래야 나중에 다시 물었을 때 검색과 연결이 쉬워지고, 블로그/FAQ로도 잘 파생됩니다.

### 3. 그냥 노트만 많이 만들면 안 되나요?
양은 늘어도 기준점과 연결이 없으면 실제 운영에선 다시 못 쓰는 경우가 많습니다.

### 4. 실제 운영에서 가장 중요한 활용법은 무엇인가요?
새 질문이 오면 관련 주제 문서를 찾고, 새 사례가 나오면 기존 축에 붙이며, 파생 후보를 계속 남기는 것입니다.

### 5. 왜 Obsidian이 잘 맞나요?
문서 간 연결과 source note 중심 파생 설계를 하기에 적합하기 때문입니다.
