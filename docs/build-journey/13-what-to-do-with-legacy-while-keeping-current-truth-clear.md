# 과거 유산은 지워야 할까 남겨야 할까

## TL;DR
과거 시스템 유산은 완전히 지워야 할 쓰레기가 아닙니다. 전환사, 비교 기준, 스킬 이관, 시행착오 설명에 필요한 역사 자산이기 때문입니다. 다만 현재 운영의 기준점은 반드시 하나여야 합니다. 핵심은 삭제가 아니라 **무엇을 active docs에 두고, 무엇을 archive/history/skill 층으로 옮길지 정리하는 거버넌스**입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

과거 유산은 남겨도 되지만,
현재 운영 판단과 사용자-facing 기준은 반드시 현재 시스템 쪽으로 하나로 세워야 합니다.

핵심은 다섯 가지입니다.
- 유산은 이해의 재료이자 전환사의 근거다.
- 현재 운영 기준은 하나여야 혼선이 줄어든다.
- 가장 위험한 상태는 과거 흔적과 현재 규칙이 같은 문서 층에 섞이는 것이다.
- archive/history/skill imports/migration notes에는 남겨도 된다.
- active docs와 현재 persona 기준 문서에선 현재 규칙이 먼저 보여야 한다.

즉,
이건 지울지 말지의 문제가 아니라 **위치와 거버넌스를 어떻게 정리할 것인가**의 문제입니다.


---

## 왜 사람은 과거 흔적을 보면 현재 규칙도 같이 흔들릴까
시스템 전환이 있으면 흔적은 남습니다.
문서, 말투, 스킬, 런타임, 설명 방식까지 다요.

문제는 이 흔적 자체가 아닙니다.
문제는 사용자가 이걸 보며 이렇게 느끼는 순간입니다.
- 지금 기준이 뭘까?
- 이 문장은 예전 설명인가, 현재 규칙인가?
- 왜 예전 감각이 아직 살아 있지?
- 무엇을 source of truth로 봐야 하지?

즉,
혼선의 본질은 과거 유산의 존재가 아니라
**현재 기준이 어디인지 선명하지 않은 상태**에 있습니다.

---

## 왜 유산을 완전히 지우는 것도 답이 아닐까
여기서 첫 번째 오해가 나옵니다.
정리를 위해 다 지우는 게 깔끔해 보일 수 있습니다.

하지만 실제로는 과거 유산에 이런 가치가 남아 있습니다.
- 역할과 감각의 역사
- 이전 운영 실험의 맥락
- 가져온 스킬과 절차
- 전환사 설명의 근거
- 실패와 시행착오 기록

즉,
유산은 현재가 아니더라도 여전히 **이해의 재료**입니다.
전부 지우면 왜 이런 구조가 생겼는지 설명할 재료도 같이 사라집니다.


---

## 그렇다면 현재 기준은 왜 하나여야 하나
두 번째 핵심은 source of truth입니다.
운영은 결국 기준점이 하나여야 합니다.

특히 이런 질문은 현재 기준으로만 답해야 합니다.
- 지금 어떤 profile이 기준인가?
- 지금 어떤 문서를 따라야 하는가?
- 지금 어떤 에이전트 구조가 운영 원칙인가?
- 지금 어떤 tone/route/rule이 유효한가?

여기에 과거 규칙이 섞이면,
운영자는 매번 번역을 해야 하고 사용자는 계속 체감 혼선을 느낍니다.

즉,
과거를 남기되 현재 기준은 반드시 하나로 세우는 게 중요합니다.

---

## 그럼 무엇을 어디에 남기면 되나
우리 기준에선 보통 이렇게 나눕니다.

### 남겨도 되는 층
- archive
- history
- skill imports
- migration notes
- comparison docs
- lesson learned docs

### 현재 운영 문서에서 빼야 하는 층
- 메인 운영 원칙 문서
- 현재 persona 기준 문서
- 현재 사용자-facing 설명
- 현재 하비/방울이/뽀동이 운영 문맥

핵심은,
유산을 지우는 게 아니라 **현재와 같은 층에 두지 않는 것**입니다.
그렇게 해야 전환사는 남고 운영 기준은 흐려지지 않습니다.


---

## 왜 이게 기술 문제가 아니라 거버넌스 문제인가
세 번째 핵심은 거버넌스입니다.
많은 혼선은 기술 에러처럼 보이지만 실제로는 기준점 설계 문제입니다.

예:
- 왜 말투가 다르지?
- 왜 문서 기준이 다르지?
- 왜 예전 시스템 감각이 자꾸 튀어나오지?
- 왜 하비가 과거 흔적처럼 느껴지지?

이런 질문의 상당수는,
실행 엔진 자체보다 **문서 층위와 규칙 층위가 섞였기 때문**입니다.

즉,
좋은 운영은 더 많은 설명을 붙이는 게 아니라
현재 규칙과 역사 설명의 층을 명확히 나누는 데서 시작됩니다.

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 현재 운영 문서는 현재 기준으로 쓴다
active doc에선 현재 규칙이 먼저 보여야 한다.

### 원칙 2. 과거는 역사/전환/비교 문서에서 설명한다
현재 규칙처럼 보이지 않게 둔다.

### 원칙 3. 유산은 남기되 위치를 옮긴다
archive/history/skill 층으로 정리한다.

### 원칙 4. 사용자-facing 설명은 현재 기준만 선명하게 유지한다
복잡성은 뒤로 숨긴다.

### 원칙 5. 삭제보다 source of truth 정리가 먼저다
무엇을 믿어야 하는지를 명확히 한다.

**유산은 지우는 게 아니라 위치를 옮기는 것이고, 현재 운영 기준은 반드시 하나로 세워야 한다.**

그래서 좋은 정리는 삭제의 미학이 아니라,
현재와 과거가 섞이지 않게 층을 나누는 거버넌스에서 나옵니다.

## FAQ
### 1. 과거 유산은 왜 완전히 지우지 않나요?
전환사, 비교 기준, 시행착오 설명에 필요한 이해 재료이기 때문입니다.

### 2. 현재 운영 기준은 어디를 봐야 하나요?
현재 active docs, current profile/docs/wiki처럼 현재 규칙이 먼저 보이는 층을 봐야 합니다.

### 3. archive/history와 active docs는 어떻게 구분하나요?
역사 설명과 현재 규칙이 같은 문서 층에 섞이지 않게 나누는 게 핵심입니다.

### 4. 왜 유산이 남아 있으면 체감 혼선이 생기나요?
과거 설명이 현재 규칙처럼 보이거나, 현재 기준점이 흐려질 수 있기 때문입니다.

### 5. 가장 먼저 바꿔야 할 습관은 무엇인가요?
이 문서가 현재 규칙인가, 역사 설명인가를 먼저 구분하는 습관입니다.
