# 에르메스 운영에서 제일 자주 나오는 질문들

## TL;DR
Hermes 운영에서 자주 나오는 질문은 겉으로는 다양해 보여도 실제로는 몇 개의 판단 축으로 모입니다. 정체성/전환, profile/기억, 역할 구조, 도구 연동, 문서/볼트, runtime/gateway 같은 축입니다. 그래서 좋은 FAQ는 답을 많이 외우게 하는 문서가 아니라 **지금 문제가 어느 레이어인가를 먼저 나누게 하는 판단 프레임**에 가깝습니다.

## 이 글의 답
답부터 말하면 이렇습니다.

Hermes 운영에서 가장 자주 나오는 질문은,
겉보기와 달리 몇 개의 반복 축으로 수렴하고,
좋은 운영은 답을 늘리는 것보다 **판단 레이어를 먼저 구분하는 것**에서 시작됩니다.

핵심은 다섯 가지입니다.
- 반복 질문은 정체성, profile/기억, 역할 구조, 도구, 문서, runtime 축으로 모인다.
- 체감 문제처럼 보여도 보통은 구조 문제다.
- FAQ는 답변 모음이 아니라 판단 기준 압축본이어야 한다.
- 헷갈릴수록 session/memory/skill/profile/runtime/source-of-truth를 먼저 나눠야 한다.
- 좋은 FAQ는 항상 더 깊은 원천 문서로 연결된다.

즉,
좋은 운영 FAQ는 “무엇이 답인가”만 말하지 않고 **어떻게 판단해야 하는가**를 같이 남깁니다.

![에르메스 운영에서 제일 자주 나오는 질문들](/assets/build-journey/most-common-hermes-ops-questions-and-how-to-think/og-card-final.png)

---

## 왜 운영 질문은 계속 비슷한 데서 반복될까
처음엔 질문이 제각각처럼 보입니다.
- 왜 같은 하비인데 느낌이 다르지?
- 왜 예전 대화를 못 찾지?
- 왜 MCP는 붙었는데 오류가 나지?
- 왜 파일은 있는데 못 읽지?
- 왜 위키를 먼저 쌓지?
- heartbeat가 보이면 뭐가 살아 있는 거지?

하지만 실무에서 보면,
이 질문들은 몇 개의 축으로 계속 모입니다.

즉,
좋은 운영은 질문마다 새 답을 만드는 게 아니라
**반복 질문을 몇 개의 레이어로 번역하는 것**에서 쉬워집니다.

---

## 실제로 가장 자주 나오는 질문 축은 무엇인가
우리 기준에선 보통 이렇게 나눕니다.

### 1. 정체성/전환 질문
- OpenClaw와 Hermes는 같은가 다른가?
- 현재 기준은 어디를 봐야 하나?

### 2. profile / 기억 질문
- 왜 같은 하비인데 느낌이 다르지?
- 왜 예전 대화를 못 찾지?

### 3. 역할 구조 질문
- 왜 하비가 메인 창구인가?
- 왜 방울이/뽀동이가 따로 필요한가?

### 4. 도구/연동 질문
- Google Workspace는 왜 자꾸 꼬이지?
- MCP는 왜 붙었는데 깨지지?

### 5. 문서/볼트 질문
- 왜 파일은 있는데 못 읽지?
- 왜 위키를 먼저 쌓지?

### 6. runtime/gateway 질문
- heartbeat가 보이면 뭐가 살아 있는 거지?
- gateway status와 실제 프로세스가 왜 다르지?

즉,
질문은 많아 보여도
**운영 판단 축은 생각보다 적습니다.**

![반복 질문은 몇 개의 축으로 모인다](/assets/build-journey/most-common-hermes-ops-questions-and-how-to-think/article-figure-01-question-clusters.png)

---

## FAQ가 답변 모음이면 왜 약할까
두 번째 핵심은 FAQ의 역할입니다.
많은 FAQ 문서는 답을 그냥 나열합니다.

하지만 Hermes 운영에선 이 방식이 약합니다.
왜냐하면 체감 문제와 구조 문제가 섞여 있기 때문입니다.

예:
- 하비가 다르게 느껴진다 → profile/state.db/session 문제일 수 있음
- 예전 대화를 못 찾는다 → memory가 아니라 session 회수 문제일 수 있음
- MCP가 깨진다 → auth/stdout/protocol 문제일 수 있음

즉,
좋은 FAQ는 “정답”보다 먼저
**지금 문제가 어느 레이어인가**를 물어야 합니다.

그래야 체감 문제를 구조 문제로 번역할 수 있습니다.

---

## 그럼 실전에서 어떤 프레임으로 보면 되나
우리 기준에선 보통 이 프레임이 제일 잘 먹힙니다.

### A. 지금 문제는 어느 레이어인가
- session인가?
- memory인가?
- skill인가?
- profile/state.db인가?
- runtime/gateway인가?
- 문서/source-of-truth인가?

### B. 현재 기준점은 어디인가
- active doc인가?
- archive/history인가?
- 현재 profile인가?
- 과거 흔적인가?

### C. 지금 필요한 건 무엇인가
- 사실 확인인가?
- 정리/재구성인가?
- 규칙 점검인가?
- 구조적 재설계인가?

핵심은,
답변을 외우기보다
**지금 무슨 종류의 문제인지 먼저 판별하는 것**입니다.

![좋은 FAQ는 판단 프레임을 준다](/assets/build-journey/most-common-hermes-ops-questions-and-how-to-think/article-figure-02-decision-frame.png)

---

## 왜 이 프레임이 실제로 유용한가
세 번째 핵심은 재사용성입니다.
이 프레임이 있으면 FAQ 하나가 블로그, 강의, 위키 허브로 바로 확장됩니다.

예:
- “왜 같은 하비인데 느낌이 다르지?” → profile/session/memory 강의 포인트
- “왜 파일은 있는데 못 읽지?” → 접근 레이어 블로그
- “왜 위키를 먼저 쌓지?” → 콘텐츠 파이프라인 문서
- “heartbeat가 보이면 뭐가 살아 있는 거지?” → runtime 운영 문서

즉,
반복 질문은 귀찮은 support backlog가 아니라
**운영 구조가 드러나는 가장 좋은 출발점**이 됩니다.

![FAQ는 위키와 블로그의 진입점이다](/assets/build-journey/most-common-hermes-ops-questions-and-how-to-think/article-figure-03-faq-to-content.png)

---

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 답보다 레이어를 먼저 본다
헷갈릴수록 session, memory, skill, profile, runtime, docs를 먼저 나눈다.

### 원칙 2. 체감보다 저장소와 런타임을 먼저 본다
느낌 문제처럼 보여도 구조 문제일 때가 많다.

### 원칙 3. 반복 질문은 FAQ로만 끝내지 않는다
원천 위키 문서와 연결한다.

### 원칙 4. FAQ는 짧게 답하되 더 깊은 문서로 이어지게 한다
허브 역할을 하게 만든다.

### 원칙 5. 반복 질문은 운영 구조가 드러나는 신호로 본다
불만 목록이 아니라 개선 입구로 본다.

---

## 결론
Hermes 운영에서 가장 자주 나오는 질문은,
사실 그렇게 많지 않습니다.

핵심은 이겁니다.
**좋은 운영 FAQ는 답을 나열하는 문서가 아니라, 판단 기준을 압축한 문서다.**

그래서 헷갈릴수록 답을 더 많이 찾기보다,
먼저 지금 문제가 어느 레이어인지 나누는 편이 훨씬 빠릅니다.

---

## FAQ
### 1. 왜 FAQ에 답만 있으면 부족한가요?
체감 문제와 구조 문제가 섞여 있어, 정답만 나열하면 재사용성이 약해지기 때문입니다.

### 2. 가장 먼저 나눠야 할 레이어는 무엇인가요?
session, memory, skill, profile/state.db, runtime/gateway, docs/source-of-truth 축입니다.

### 3. 왜 반복 질문이 중요한가요?
운영 구조가 어디서 자주 헷갈리는지 보여주는 가장 좋은 신호이기 때문입니다.

### 4. FAQ는 어디까지 확장할 수 있나요?
블로그, 강의, 위키 허브 진입점, 체크리스트까지 확장할 수 있습니다.

### 5. 가장 먼저 바꿔야 할 습관은 무엇인가요?
바로 답을 찾기보다 먼저 “이 문제는 어느 레이어인가?”를 묻는 습관입니다.

## 내부 링크 포인트

- 허브 문서는 Topic - Hermes Agent Ops로 연결한다.
- profile 경계는 Topic - profile 경계와 state db를 실무에서 어떻게 봐야 하는가로 연결한다.

## CTA
요즘 운영 질문이 자꾸 늘어나는 느낌이라면,
답을 더 붙이기보다 먼저 **판단 레이어를 정리한 FAQ 프레임**부터 세우는 편이 좋습니다.

## 소셜 카피 초안
### 옵션 1
좋은 운영 FAQ는 답을 나열하는 문서가 아니다. 헷갈릴수록 답을 늘리기보다, 지금 문제가 어느 레이어인지 먼저 나누는 판단 기준이 더 중요하다.

### 옵션 2
반복 질문은 귀찮은 support backlog가 아니라 운영 구조가 드러나는 가장 좋은 출발점이다. FAQ는 정답 모음보다 판단 프레임 압축본에 가까워야 한다.
