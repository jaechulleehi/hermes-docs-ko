# 멀티봇 스레드는 왜 쉽게 시끄러워질까

## TL;DR
멀티봇 스레드는 자유로워 보입니다. 여러 역할형 봇이 한 스레드 안에서 바로 도와줄 수 있으니까요. 그런데 규칙 없이 두면 금방 시끄러워집니다. 누가 언제 답해야 하는지 예측이 안 되고, 사용자는 스레드 주도권을 잃고, 결과도 섞이기 쉽기 때문입니다. 핵심은 봇 수가 아니라 **응답 조건을 선명하게 만드는 것**, 그리고 여러 봇이 있어도 사용자는 누가 언제 말하는지 예측 가능해야 한다는 점입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

멀티봇 스레드가 쉽게 시끄러워지는 이유는,
봇이 많아서가 아니라 **누가 언제 응답해야 하는지 규칙이 불명확하기 때문**입니다.

핵심은 다섯 가지입니다.
- 기본 창구는 하나여야 한다.
- 역할형 봇은 명시 호출일 때만 반응하는 편이 낫다.
- 두 개 이상 봇이 참여 중이면 멘션 기반으로 제한하는 게 안정적이다.
- 자유응답을 늘릴수록 역할 충돌과 사용자 피로가 커진다.
- 좋은 호출 규칙은 기술 규칙이자 사용자 경험 설계다.

즉,
멀티봇 운영의 핵심은 더 많이 말하게 하는 게 아니라 **더 예측 가능하게 말하게 하는 것**입니다.


---

## 왜 멀티봇은 처음엔 멋져 보이는데 나중엔 피곤해질까
처음엔 장점이 크게 보입니다.
- 역할별로 바로 부를 수 있다.
- 조사형/정리형/총괄형이 한 스레드에 다 있을 수 있다.
- 필요한 역할이 바로 튀어나오면 편해 보인다.

그런데 실제로는 이런 일이 생깁니다.
- 하비에게 말했는데 방울이도 같이 답한다.
- 한 번 멘션했더니 이후엔 계속 자동 반응한다.
- 두 봇이 동시에 자기 몫이라고 생각한다.
- 사용자는 누가 응답할지 예측하기 어렵다.

즉,
문제는 봇 수 자체가 아니라
**응답 조건이 불명확한 것**입니다.

---

## 왜 호출 규칙이 없으면 금방 혼선이 커지나
여기서 첫 번째 핵심이 나옵니다.
호출 규칙은 단순 예절이 아닙니다.
이건 시스템 안정성을 위한 설계입니다.

좋은 호출 규칙이 있으면:
- 예측 가능성이 생긴다.
- 스레드가 덜 시끄럽다.
- 역할 충돌이 줄어든다.
- 사용자가 누구를 기대해야 하는지 명확해진다.

반대로 규칙이 없으면:
- 자동응답이 겹친다.
- 결과가 섞인다.
- 사용자 피로가 커진다.
- 멀티봇 구조가 귀찮은 장난감처럼 느껴진다.

즉,
좋은 멀티봇 운영은 기술 과시보다
**응답 조건 설계**가 먼저입니다.


---

## 가장 좋은 기본 원칙은 무엇인가
우리 기준에선 보통 이 조합이 가장 안정적입니다.

### 1. 기본 창구는 하비
특별히 다른 봇을 부르지 않았다면 하비가 기본 응답 주체가 됩니다.

### 2. 역할형 봇은 명시 호출형
방울이/뽀동이 같은 역할형 봇은 이름 호출이나 멘션이 있을 때만 반응하게 합니다.

### 3. 봇이 둘 이상 참여 중이면 explicit mention only
이 규칙이 특히 중요합니다.
- single-bot thread = 자연스러운 follow-up 허용
- multi-bot thread = explicit mention only

핵심은,
자유응답을 줄이는 대신
**기대와 주도권을 사용자 쪽에 남겨두는 것**입니다.

---

## 왜 이 규칙이 하비 구조와 잘 맞나
두 번째 핵심은 메인 창구 구조와의 결합입니다.
하비 메인 창구의 목표는 사용자가 내부 복잡성을 직접 관리하지 않게 하는 것입니다.
멀티봇 호출 규칙도 같은 문제를 풉니다.

- 기본은 하비
- 필요할 때만 역할형 봇 호출
- 여러 봇이 있으면 명시적으로 지목

즉,
호출 규칙은 하비 구조의 UX 버전이라고 볼 수 있습니다.
복잡성은 뒤로 숨기고 사용자는 전진만 하게 만드는 방식입니다.


---

## 어떤 설계가 실제로 잘못되기 쉬운가
세 번째 핵심은 실패 패턴입니다.
이런 구조는 금방 피곤해집니다.

- 한 번 멘션되면 그 스레드에서 계속 자동 반응
- 역할형 봇도 기본 follow-up에 계속 반응
- 규칙이 코드에는 있는데 사용자에게 안 보임
- 모든 스레드에서 무조건 멘션만 강제

즉,
좋은 호출 규칙은 너무 느슨해도 문제고,
너무 경직돼도 문제입니다.

그래서 보통은
**기본은 메인 창구, 다봇 환경에선 명시 호출**이 제일 안정적입니다.

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 기본 창구는 하비
사용자는 일단 하비만 부르면 된다.

### 원칙 2. 역할형 봇은 명시 호출형
방울이/뽀동이는 이름을 불렀을 때만 반응한다.

### 원칙 3. 두 개 이상 봇이 있는 스레드는 멘션 기반으로 제한
그 순간부터 자연 follow-up보다 explicit mention이 낫다.

### 원칙 4. 규칙은 사용자에게도 짧게 보여야 한다
체감이 예측 가능해야 한다.

### 원칙 5. 혼선이 커지면 새 스레드로 분기한다
한 스레드에서 모든 걸 해결하려 하지 않는다.

## FAQ
### 1. 왜 방울이/뽀동이는 평소엔 조용한가요?
기본 창구 경험을 유지하고 역할형 과잉 반응을 막기 위해서입니다.

### 2. 왜 두 봇이 있는 스레드에서는 멘션이 더 중요하나요?
응답 기대를 선명하게 해야 역할 충돌과 혼선이 줄어들기 때문입니다.

### 3. 왜 한 번 부른 봇이 계속 자동 반응하면 안 좋은가요?
처음엔 편해 보여도 봇이 여러 개 섞이면 금방 혼선과 피로가 커지기 때문입니다.

### 4. 메인 창구가 있으면 멀티봇은 왜 필요한가요?
사용자에겐 단일 입구를 주되, 내부적으론 역할 분리 품질을 가져가기 위해서입니다.

### 5. 가장 먼저 바꿔야 할 습관은 무엇인가요?
멀티봇일수록 자유응답보다 호출 규칙을 먼저 설계하는 습관입니다.
