# 하비가 언제 직접 처리하고 언제 위임해야 할까

## TL;DR
메인 창구의 핵심은 모든 걸 직접 하는 데 있지 않습니다. 오히려 중요한 건 무엇을 바로 잡아야 하고, 무엇을 조사형/정리형 에이전트로 태워야 하는지 빨리 판단하는 것입니다. Harvey가 다 직접 처리하면 병목이 생기고, 반대로 너무 쉽게 위임하면 결과 통합과 품질 책임이 약해집니다. 그래서 핵심은 **직접 처리 기준과 위임 기준을 분명히 나누는 것**입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

하비는,
단순하고 결정이 선명한 일은 직접 잡고,
조사·정리·반복 구조화가 필요한 일은 위임하되 최종 책임은 계속 가져가야 합니다.

핵심은 다섯 가지입니다.
- 창구 해석과 최종 통합은 하비 몫이다.
- 빠른 판단이 필요한 단순 작업은 직접 처리하는 편이 낫다.
- 조사/탐색/근거 수집은 조사형 위임이 낫다.
- 구조화/보고서/초안 작성은 정리형 위임이 낫다.
- 위임의 핵심은 던지는 게 아니라 다시 받아 검수·통합하는 것이다.

즉,
좋은 메인 창구는 많이 하는 창구가 아니라 **잘 나누고 끝까지 책임지는 창구**입니다.

## 왜 메인 창구는 다 하면 안 되고, 다 넘겨도 안 될까
하비가 다 직접 처리하면 이런 문제가 생깁니다.
- 병목이 생긴다.
- 조사와 정리의 전문성이 약해진다.
- 운영 속도가 떨어진다.

반대로 너무 쉽게 위임하면 이런 문제가 생깁니다.
- 사용자는 중간 산출물을 너무 많이 본다.
- 결과가 흩어진다.
- 최종 책임이 흐려진다.

즉,
메인 창구의 핵심은 노동량이 아니라
**분배 기준과 책임 구조**입니다.

---

## 직접 처리 기준은 무엇인가
여기서 첫 번째 핵심이 나옵니다.
보통 이런 일은 하비가 직접 잡는 편이 빠릅니다.

- 단순 판단이나 우선순위 정리
- 짧은 확인과 응답
- 사용자 대화 흐름 유지
- 이미 기준이 선명한 작업
- 위임 오버헤드가 더 큰 작업

즉,
직접 처리는 “하비가 할 수 있느냐”보다
**지금 직접 하는 게 더 빠르고 덜 흔들리느냐**가 기준입니다.

![직접 처리 기준](/assets/build-journey/when-harvey-should-handle-directly-vs-delegate/article-figure-01-direct-handle-rules.png)

---

## 위임 기준은 무엇인가
두 번째 핵심은 위임 적합성입니다.
보통 이런 일은 위임하는 쪽이 낫습니다.

### 조사형 위임이 맞는 일
- 탐색
- 사실 확인
- 근거 수집
- 비교 후보 정리

### 정리형 위임이 맞는 일
- 문서화
- 보고서 초안
- 발표용 재구성
- 구조화된 글쓰기

핵심은,
위임이 책임 회피가 아니라
**역할 특화 품질을 끌어오는 수단**이어야 한다는 점입니다.

---

## 왜 위임의 핵심은 다시 받아 검수하는 데 있나
세 번째 핵심은 통합 책임입니다.
좋지 않은 위임은 “태우고 끝”입니다.
좋은 위임은 “태우고 다시 받아 검수·통합”입니다.

하비가 계속 가져가야 하는 일:
- 목적 해석
- 출력 기대 정리
- 리스크 표시
- 결과 검수
- 최종 사용자-facing 통합

즉,
하비의 역할은 스스로 다 하는 게 아니라
**끝까지 품질과 맥락을 묶는 것**입니다.

![위임의 핵심은 검수와 통합이다](/assets/build-journey/when-harvey-should-handle-directly-vs-delegate/article-figure-02-delegate-and-integrate.png)

## 그럼 실전에서는 어떤 순서로 판단하면 되나
우리 기준에선 보통 이 순서가 잘 맞습니다.

### 1. 지금 이 일은 창구 해석이 핵심인가
그렇다면 하비가 먼저 잡는다.

### 2. 근거 수집/탐색이 큰 비중인가
그렇다면 조사형 위임이 맞다.

### 3. 구조화/작성/문서화가 큰 비중인가
그렇다면 정리형 위임이 맞다.

### 4. 위임 오버헤드가 직접 처리보다 큰가
그렇다면 직접 잡는 편이 낫다.

### 5. 결과를 다시 검수·통합해야 하는가
그렇다면 최종 책임은 계속 하비가 가진다.

즉,
좋은 메인 창구의 힘은 실행력이 아니라
**직접 처리와 위임의 경계를 빠르게 가르는 판단력**입니다.

![직접 처리 vs 위임 판단 순서](/assets/build-journey/when-harvey-should-handle-directly-vs-delegate/article-figure-03-routing-check-order.png)

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 창구 해석과 최종 통합은 하비 몫
### 원칙 2. 조사/정리 역할은 특화 위임
### 원칙 3. 위임은 던지는 게 아니라 다시 받아 검수하는 것
### 원칙 4. 위임 오버헤드가 크면 직접 처리
### 원칙 5. 사용자는 내부 복잡성을 보지 않게 만든다

## 결론
하비가 언제 직접 처리하고 언제 위임해야 하는가는,
단순 분업 문제가 아닙니다.

핵심은 이겁니다.
**좋은 메인 창구는 모든 걸 직접 하는 창구가 아니라, 무엇을 직접 잡고 무엇을 위임할지 빠르게 고르되 최종 책임은 끝까지 지는 창구다.**

## FAQ
### 1. 하비가 다 직접 하면 더 빠른 것 아닌가요?
짧은 일은 그럴 수 있지만, 조사·정리 전문성이 필요한 일까지 다 직접 하면 병목이 생깁니다.

### 2. 위임하면 책임도 같이 넘어가나요?
아닙니다. 최종 해석과 사용자-facing 통합 책임은 계속 하비가 가집니다.

### 3. 가장 먼저 봐야 할 기준은 무엇인가요?
이 일이 창구 해석, 조사, 정리 중 어디 비중이 가장 큰지 보는 것입니다.
