# 복구 플레이북은 왜 문서보다 순서가 중요할까

## TL;DR
복구 문서는 길수록 좋아 보일 수 있습니다. 체크리스트도 많고, 명령도 많고, 경우의 수도 많으니까요. 하지만 실제 복구 실패는 문서 부족보다 **순서를 잘못 잡아서** 생기는 경우가 훨씬 많습니다. status만 보고 재시작하거나, profile/runtime 경계를 확인하기 전에 전체를 다시 띄우거나, autostart 충돌을 나중에 보는 식이 대표적입니다. 그래서 좋은 복구 플레이북의 핵심은 정보량보다 **무엇을 어떤 순서로 볼 것인가**입니다.

## 이 글의 답
답부터 말하면 이렇습니다.

복구 플레이북이 문서보다 순서가 중요한 이유는,
문제의 성질을 빨리 구분하지 못하면 같은 명령도 오히려 혼선을 키우기 때문입니다.

핵심은 다섯 가지입니다.
- 복구는 “다시 켜기”보다 “어느 층이 죽었는지 구분하기”가 먼저다.
- status보다 실제 process를 먼저 봐야 한다.
- profile/runtime 경계를 확인하기 전 전체 재시작은 위험하다.
- autostart/legacy 충돌은 복구 후반이 아니라 중간 점검 대상이다.
- 좋은 복구는 재시작이 아니라 **정확한 순서의 진단과 최소 침습 복구**다.

즉,
복구 플레이북의 힘은 긴 문서보다 **잘못된 순서를 막는 구조**에서 나옵니다.


---

## 왜 복구는 문서가 있어도 실패하나
복구 문서가 없는 팀도 문제지만,
문서가 있는데도 실패하는 팀도 많습니다.

왜냐하면 대부분의 실패는 이런 식이기 때문입니다.
- status만 보고 재시작한다.
- 실제 프로세스를 안 본다.
- 다른 profile까지 같이 건드린다.
- old runtime이 살아 있는 걸 못 본다.
- autostart가 다시 old system을 띄운다.

즉,
문제가 명령 부족이 아니라
**진단 순서 오판**인 경우가 많습니다.

---

## 왜 process를 먼저 봐야 하나
여기서 첫 번째 핵심이 나옵니다.
복구는 늘 “실제로 무엇이 살아 있는가”부터 시작해야 합니다.

status는 참고값일 뿐이고,
실제 process, PID, 어떤 profile로 실행됐는지가 먼저입니다.

이 순서를 놓치면:
- 이미 살아 있는 걸 또 띄우고
- 죽은 줄 알았던 걸 놓치고
- 중복 runtime을 만들고
- 더 큰 혼선을 스스로 추가하게 됩니다.

즉,
복구의 시작점은 명령 실행이 아니라
**관측 정확도**입니다.


---

## 왜 profile/runtime 경계를 먼저 확인해야 하나
두 번째 핵심은 경계입니다.
게이트웨이 복구에서 자주 생기는 오판은,
현재 끊긴 profile과 다른 살아 있는 profile을 구분하지 못하는 데서 나옵니다.

예:
- harvey만 문제인데 전체를 다 재시작
- old runtime을 현재 runtime으로 착각
- state file과 실제 실행 주체를 섞어 봄

즉,
복구는 “꺼졌는가”보다
**무엇이, 어디서, 어느 profile로, 어떤 상태에서 끊겼는가**를 먼저 나눠야 합니다.

이 구분 없이 재시작만 하면,
복구가 아니라 혼선 증폭이 됩니다.

---

## 왜 복구는 최소 침습이 중요한가
세 번째 핵심은 범위입니다.
복구가 잘 안 되는 팀일수록 자꾸 전체 재시작에 기대기 쉽습니다.

하지만 좋은 복구는 보통:
- 끊긴 profile만 살리고
- 살아 있는 다른 프로필은 건드리지 않고
- 필요한 층만 최소한으로 복구하고
- 마지막에 검증까지 붙입니다.

즉,
복구의 목적은 “다시 켜기 성공”이 아니라
**운영 전체를 덜 흔들고 필요한 부분만 정확히 살리는 것**입니다.


---

## 그럼 좋은 복구 플레이북은 어떻게 생겨야 하나
우리 기준에선 보통 이 순서가 핵심입니다.

### 1. 증상 분류
응답 없음 / 이상 응답 / 특정 profile 문제 / status-process 불일치

### 2. 실제 프로세스 확인
gateway process, PID, 실행 profile, 중복 runtime 여부

### 3. profile / runtime 경계 확인
어느 profile이 실제 죽었는지 구분

### 4. 최소 복구 시도
끊긴 profile만 살리는 쪽 우선

### 5. autostart / legacy 충돌 점검
다시 old runtime을 띄우는 요인 확인

### 6. 복구 후 검증
실제 응답, 기대 profile, 중복 runtime 여부 확인

핵심은,
플레이북이 많은 걸 설명하는 문서가 아니라
**잘못된 복구 순서를 막아주는 진단 구조**여야 한다는 점입니다.

## FAQ
### 1. 왜 status만 보고 복구하면 안 되나요?
status는 참고값일 뿐 실제 process와 어긋날 수 있기 때문입니다.

### 2. 전체 재시작이 왜 위험한가요?
살아 있는 다른 profile까지 흔들고 중복 runtime을 만들 수 있기 때문입니다.

### 3. 가장 먼저 바꿔야 할 습관은 무엇인가요?
재시작 전에 실제 process와 profile을 먼저 보는 습관입니다.
