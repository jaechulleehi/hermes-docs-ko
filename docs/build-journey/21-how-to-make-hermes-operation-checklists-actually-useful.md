# 에르메스 운영 체크리스트는 어떻게 써야 실제로 도움이 될까

## TL;DR
운영 체크리스트는 문서를 줄이기 위한 요약본이 아닙니다. 실제 문제 상황에서 무엇부터 확인할지 순서를 강제하는 운영 도구입니다. 그래서 좋은 체크리스트는 모든 걸 설명하지 않고, 자주 반복되는 profile·runtime·memory·gateway·handoff 같은 축을 짧게 나눠서 바로 꺼내 쓸 수 있게 만들어야 합니다.

## 이 글의 답
답부터 말하면 이렇습니다.
에르메스 운영 체크리스트가 실제로 도움이 되려면, 설명을 많이 담는 문서가 아니라 판단 순서를 빠르게 복구하는 도구여야 합니다.
핵심은 다섯 가지입니다.
- 체크리스트는 본문 문서를 대체하지 않고, 본문으로 다시 붙게 만드는 입구여야 한다.
- 항목은 profile, runtime, memory/skill, gateway, handoff처럼 반복되는 운영 축으로 나뉘어야 한다.
- 길게 설명하기보다 무엇을 먼저 볼지 순서를 줘야 한다.
- 새 시행착오가 생기면 새 문서를 늘리기보다 기존 체크리스트 축에 붙이는 편이 강하다.
- 좋은 체크리스트는 운영 속도를 올리지만, 나쁜 체크리스트는 오히려 읽을 문서만 하나 더 만든다.
즉, 체크리스트의 가치는 정보량보다 재접속 속도와 재판단 속도에 있습니다.

## 왜 체크리스트는 많은데도 실제로는 잘 안 쓰일까

체크리스트가 도움이 안 되는 가장 흔한 이유는, 설명서처럼 길어지기 때문입니다.

운영 중에는 차분히 긴 문서를 읽을 시간이 없습니다. 지금 profile이 맞는지, runtime이 살아 있는지, memory 문제인지 skill 문제인지 같은 것을 짧게 다시 붙어야 합니다.

그래서 체크리스트가 문서 요약본처럼 보이기 시작하면 오히려 가치가 떨어집니다.

급할 때 펼쳐도 무엇부터 볼지 눈에 안 들어오고, 항목이 너무 많아 실제 순서가 흐려지고, 결국 다시 원문 문서로 돌아가거나 감으로 처리하게 됩니다.

즉, 체크리스트의 실패는 보통 내용 부족이 아니라 형태 오판에서 시작합니다.

![운영 체크리스트가 실패하는 이유](/assets/build-journey/how-to-make-hermes-operation-checklists-actually-useful/article-figure-01-how-to-make-hermes-operation-checklists-actually-useful.png)

## 좋은 체크리스트는 무엇을 먼저 하게 만드나

에르메스 운영에서는 문제가 대개 몇 개 축으로 반복됩니다.

대표 축은 profile 전환, gateway/runtime 이상, heartbeat나 legacy 혼선, memory / session / skill 경계 혼동, handoff 품질 부족, 블로그/위키 파생 기준 미정입니다.

좋은 체크리스트는 이 반복 축을 기준으로 정리합니다.

이렇게 하면 문제가 생길 때마다 새 문서를 찾는 대신, 먼저 어느 축의 문제인지부터 구분할 수 있습니다.

체크리스트의 힘은 모든 답을 들고 있는 데 있지 않고, 문제의 층을 빨리 분류하게 만드는 데 있습니다.

![좋은 체크리스트의 기본 축](/assets/build-journey/how-to-make-hermes-operation-checklists-actually-useful/article-figure-02-how-to-make-hermes-operation-checklists-actually-useful.png)

## 체크리스트는 왜 짧아야 하나

짧아야 한다는 말은 대충 만들라는 뜻이 아닙니다.

설명은 원문 문서에 남기고, 체크리스트는 순서만 남기라는 뜻입니다.

예를 들어 profile 전환 문제라면 active profile 확인 → state.db/session 경로 확인 → skill 차이 확인 → AGENTS/SOUL/memory 차이 확인 → config/auth 차이 확인 정도면 충분합니다.

여기서 각 항목의 배경 설명은 Topic이나 Session으로 보내면 됩니다.

즉, 체크리스트는 왜 그런가를 길게 말하는 곳이 아니라, 무엇부터 다시 붙을 것인가를 잡는 곳입니다.

![짧은 체크리스트가 실제로 강한 이유](/assets/build-journey/how-to-make-hermes-operation-checklists-actually-useful/article-figure-03-how-to-make-hermes-operation-checklists-actually-useful.png)

## 그럼 체크리스트는 어떻게 운영해야 하나

우리 기준에서 좋은 운영 방식은 이렇습니다.

반복되는 문제 축별로 분리하고, 한 번에 5개 안팎의 항목으로 유지하고, 새 시행착오는 새 문서를 늘리기보다 기존 체크리스트 축에 붙입니다.

또 짧은 체크리스트에서 더 깊게 봐야 할 때 Topic 원문으로 바로 넘어갈 수 있게 원문 링크를 함께 둬야 합니다.

체크리스트 자체도 운영 자산으로 봐야 합니다. 강의, FAQ, 복구 플레이북, 온보딩 문장으로 다시 파생될 수 있기 때문입니다.

결국 좋은 체크리스트는 짧지만 얕지 않고, 빠르지만 연결이 끊기지 않는 구조입니다.

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 체크리스트는 본문을 대체하지 않는다.
### 원칙 2. 설명보다 순서를 먼저 준다.
### 원칙 3. 문제 축이 반복되면 축별 체크리스트로 묶는다.
### 원칙 4. 새 시행착오는 새 문서보다 기존 체크리스트 축에 먼저 붙인다.
### 원칙 5. 운영 속도를 높이지 못하는 체크리스트는 다시 줄인다.

## FAQ
### 1. 체크리스트가 길면 더 안전한 것 아닌가요?
급한 상황에서는 길수록 실제 사용성이 떨어집니다. 설명은 본문에 두고, 체크리스트는 순서를 남기는 편이 더 안전합니다.
### 2. 체크리스트와 위키 문서는 무엇이 다른가요?
위키 문서는 이유와 배경을 설명하고, 체크리스트는 실제 운영 중 무엇부터 볼지 빠르게 붙게 만드는 도구입니다.
### 3. 새 문제가 생길 때마다 체크리스트를 새로 만들어야 하나요?
비슷한 축의 문제라면 새 문서보다 기존 체크리스트 축에 붙이는 편이 더 강합니다.
