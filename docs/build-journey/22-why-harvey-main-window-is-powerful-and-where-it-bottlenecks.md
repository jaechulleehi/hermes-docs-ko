# 하비 메인 창구 구조는 왜 강력하고 어디서 병목이 생길까

## TL;DR
메인 창구 구조의 강점은 사용자가 누구를 먼저 불러야 할지 고민하지 않아도 된다는 점입니다. 내부적으로는 조사형·정리형·판단형 역할을 분리하면서도, 밖에서는 하나의 서비스처럼 보이게 만들 수 있습니다. 하지만 하비가 위임만 하고 통합하지 않거나, 검수 품질이 약해지면 그 강점은 곧 병목이 됩니다.

## 이 글의 답
답부터 말하면 이렇습니다.
하비 메인 창구 구조가 강력한 이유는 사용자에게는 단순함을 주고, 내부에는 정교한 역할 분리를 숨길 수 있기 때문입니다.
반대로 병목이 생기는 이유도 분명합니다. 모든 요청이 하비를 거치기 때문에, 하비가 위임·검수·통합을 제대로 못하면 전체 체감 속도와 품질이 한 번에 흔들립니다.
핵심은 다섯 가지입니다.
- 메인 창구는 진입 장벽을 낮춘다.
- 최종 책임이 선명해진다.
- 내부 역할 분리와 외부 일관성을 동시에 잡을 수 있다.
- 하지만 검수와 통합이 약해지면 전달 구조가 아니라 병목 구조가 된다.
- 메인 창구의 핵심은 다 처리하는 것이 아니라 끝까지 책임지는 것이다.

## 왜 메인 창구가 필요한가

멀티에이전트 시스템은 생각보다 쉽게 피곤해집니다.

사용자가 매번 누구를 먼저 부를지, 지금 필요한 게 조사인지 정리인지, 서로 다른 답이 나오면 누가 최종 책임자인지까지 직접 판단해야 하면 금방 지칩니다.

메인 창구는 이 피로를 줄이는 장치입니다.

즉, 하비를 메인 창구로 둔다는 건 내부 구조를 없앤다는 뜻이 아니라, 사용자에게 운영 복잡성을 전가하지 않겠다는 선택입니다.

![메인 창구가 필요한 이유](/assets/build-journey/why-harvey-main-window-is-powerful-and-where-it-bottlenecks/article-figure-01-why-harvey-main-window-is-powerful-and-where-it-bottlenecks.png)

---

## 왜 이 구조는 강한가

이 구조가 강한 이유는 세 가지가 동시에 성립하기 때문입니다.

첫째, 사용자는 하비만 부르면 되므로 경험이 단순해집니다.

둘째, 조사 실패든 정리 실패든 최종 보고 책임이 하비에게 모이므로 품질 기준이 선명해집니다.

셋째, 내부적으로는 방울이·뽀동이처럼 역할을 분리해 전문성을 유지하면서도, 밖에서는 하나의 팀처럼 보일 수 있습니다.

즉, 메인 창구의 강점은 편의성이 아니라 단순한 UX와 선명한 책임 구조를 동시에 만든다는 점입니다.

![메인 창구 구조의 장점](/assets/build-journey/why-harvey-main-window-is-powerful-and-where-it-bottlenecks/article-figure-02-why-harvey-main-window-is-powerful-and-where-it-bottlenecks.png)

## 어디서 병목이 생기나

강점이 곧 한계이기도 합니다.

모든 요청이 하비를 지나가면 하비의 판단 속도, 검수 품질, 통합 역량이 전체 시스템의 병목 지점이 됩니다.

특히 병목은 동시에 여러 작업이 몰릴 때, 중간 산출물 품질이 낮아 검수 시간이 길어질 때, 하비가 직접 처리와 위임 사이에서 오래 망설일 때, 위임만 하고 최종 통합을 하지 않을 때 잘 드러납니다.

이때 사용자는 내부 사정을 모르기 때문에, 단순히 왜 이렇게 느리지로 체감하기 쉽습니다.

![메인 창구가 병목이 되는 순간](/assets/build-journey/why-harvey-main-window-is-powerful-and-where-it-bottlenecks/article-figure-03-why-harvey-main-window-is-powerful-and-where-it-bottlenecks.png)

## 그럼 어떻게 운영해야 하나

메인 창구 구조를 지속 가능하게 만들려면 몇 가지 기준이 필요합니다.

조사/탐색/근거 수집은 방울이, 정리/문서화/초안화는 뽀동이, 우선순위 판단과 최종 통합은 하비로 고정해야 합니다.

또 하비는 위임 결과를 받아 정확성·구조·사용자 적합성을 검수한 뒤 전달해야 합니다.

필요할 때는 왜 이런 흐름으로 갔는지 짧게 설명할 수 있어야 하고, 병목이 생기면 개인보다 흐름 설계를 먼저 봐야 합니다.

그래야 메인 창구가 편한 얼굴 뒤의 불투명한 병목이 아니라, 신뢰할 수 있는 운영 구조가 됩니다.

## 우리가 실제로 세운 운영 원칙
### 원칙 1. 사용자는 하비만 부르면 된다.
### 원칙 2. 하비는 위임만 하지 않고 최종 통합까지 책임진다.
### 원칙 3. 역할 경계가 흐려지면 메인 창구의 장점도 약해진다.
### 원칙 4. 병목이 생기면 개인 능력보다 흐름 설계를 먼저 본다.
### 원칙 5. 단순한 UX를 만들수록 내부 기준은 더 정교해야 한다.

## FAQ
### 1. 왜 사용자가 방울이와 뽀동이를 직접 부르면 안 되나요?
필요한 경우 직접 호출도 가능하지만, 기본적으로는 하비가 메인 창구가 되는 편이 진입 장벽과 책임 구조를 더 단순하게 만듭니다.
### 2. 메인 창구가 있으면 왜 더 느릴 수 있나요?
모든 요청이 하비를 지나가기 때문에 위임과 검수 품질이 약하면 그 지점이 병목이 됩니다.
### 3. 메인 창구의 핵심은 무엇인가요?
다 처리하는 것이 아니라 끝까지 책임지는 것입니다.
