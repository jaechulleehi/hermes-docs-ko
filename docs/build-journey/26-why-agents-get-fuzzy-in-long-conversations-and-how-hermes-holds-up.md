# 긴 대화가 쌓일수록 에이전트는 왜 흐려지고 Hermes는 어떻게 버티나

## TL;DR
컨텍스트 부패는 긴 대화가 쌓이면 피하기 어려운 문제입니다. 중요한 기준이 대화 중간에 묻히고, 요약이 반복되며, 예전 판단 이유가 손실되기 때문입니다. 다만 Hermes는 이 문제를 그냥 긴 버퍼로 버티지 않습니다. 자동 컨텍스트 압축, 세션 분리, memory, session_search, skill 같은 장치를 이미 갖고 있습니다. 그래서 일반적인 긴 채팅형 에이전트보다 훨씬 덜 무너집니다. 다만 자동 장치만으로 끝나지는 않습니다. 중요한 결정은 문서로 올리고, 반복 작업은 skill로 고정하고, 긴 스레드는 handoff 후 새 세션으로 넘기는 운영 원칙이 함께 있어야 합니다.

## 이 글의 답
답부터 말하면 이렇습니다.

긴 대화와 세션 누적이 많아질수록 에이전트는 당연히 흐려질 수 있습니다. 이건 특정 모델만의 문제가 아니라, **너무 많은 맥락을 한 번에 붙들고 판단해야 하는 시스템의 구조적 한계**에 가깝습니다.

다만 Hermes는 이걸 그냥 감수하도록 만들지 않았습니다. 핵심은 다섯 가지입니다.
- 긴 대화를 자동으로 압축하는 context compression이 있다.
- 압축 후에는 세션을 분리해 한 세션이 끝없이 비대해지지 않게 한다.
- 장기 기억은 memory에, 과거 회수는 session_search에 맡겨 한 버퍼에 다 쌓지 않는다.
- 반복 절차는 skill로 외부화해 대화 기억 의존도를 낮춘다.
- 그래서 일반적인 채팅형 운영보다 컨텍스트 부패에 훨씬 강하다.

하지만 동시에 여기서 멈추면 안 됩니다.
**Hermes는 잘 설계되어 있지만, 오래 안정적으로 쓰려면 운영 원칙까지 같이 가져가야 합니다.**

## 왜 긴 대화는 결국 흐려질까
사람들은 보통 모델이 똑똑해지면 긴 대화도 계속 선명하게 따라갈 거라고 기대합니다.
그런데 실전에서는 반대 경험을 더 자주 합니다.

- 초반에는 선명하던 기준이 중간쯤 가면 덜 또렷해진다.
- 전에 합의한 운영 원칙이 뒤로 밀린다.
- 비슷한 말을 반복하게 된다.
- 중요한 근거는 남아 있는데 판단 이유가 흐려진다.

이건 단순히 모델이 게을러서가 아닙니다.
대화가 길어질수록 컨텍스트에는 세 가지가 같이 쌓입니다.

첫째, 중요한 정보와 덜 중요한 정보가 같은 평면에 섞입니다.
둘째, 오래된 정보는 요약이나 압축을 거치며 점점 손실됩니다.
셋째, 실제로 다시 써야 할 정보가 무엇인지 분리되지 않으면, 시스템은 전체를 들고 가려다 오히려 핵심을 놓칩니다.

즉 컨텍스트 부패는 “기억력 부족”보다 **정보 구조의 붕괴**에 더 가깝습니다.

---

## 컨텍스트 부패는 어디서 시작되나
실제로는 네 가지 지점에서 많이 시작됩니다.

### 1. 중요한 결정이 채팅 중간에만 남아 있을 때
운영 원칙이나 중요한 판단이 문서나 memory로 승격되지 않고 대화 중간에만 남아 있으면, 시간이 갈수록 회수 품질이 떨어집니다.

### 2. 반복 압축이 누적될 때
요약은 유용하지만 본질적으로 손실 압축입니다. 한 번은 버틸 수 있어도 여러 번 반복되면 뉘앙스와 근거가 사라집니다.

### 3. 과거 회수와 장기 규칙을 같은 층에 기대할 때
과거 대화는 session 계층의 일이고, 오래 유지할 선호와 규칙은 memory의 일입니다. 이 둘을 한 통으로 기대하면 무엇이 빠졌는지 진단도 어려워집니다.

### 4. 절차가 대화 속에만 머물 때
잘 풀었던 방법이 skill로 승격되지 않으면 비슷한 작업을 매번 새로 설명해야 합니다. 이 체감은 쉽게 “기억이 약해졌다”로 번역됩니다.

![컨텍스트 부패는 어디서 시작되나](/assets/build-journey/why-context-rot-happens-and-how-hermes-reduces-it/article-figure-01-context-rot-starts.png)

## Hermes는 이 문제를 어떻게 줄이도록 설계됐나
여기서 Hermes가 강합니다. Hermes는 긴 대화를 그냥 계속 버퍼에 쌓아두는 방식이 아닙니다.

### 1. 자동 컨텍스트 압축
Hermes는 긴 대화가 임계치에 가까워지면 ContextCompressor로 중간 대화를 요약 압축합니다.

핵심은 무조건 자르는 게 아니라:
- 앞부분 일부를 보호하고
- 최근 tail을 보호하고
- 오래된 큰 tool output은 먼저 정리하고
- 중간 대화를 구조화 요약으로 바꾼다는 점입니다.

즉 “전부 버티기”가 아니라 **핵심만 남기고 중간을 접는 구조**입니다.

### 2. 압축 후 세션 분리
Hermes는 압축 후에도 같은 세션을 무한히 끌고 가지 않습니다. 세션을 새로 이어서 만들고, 이전 세션과 lineage를 남깁니다.

이게 중요한 이유는 단순합니다.
- 한 세션이 끝없이 비대해지지 않는다.
- 검색과 회고가 쉬워진다.
- 긴 스레드 하나가 전체 품질을 끌어내리는 현상을 줄인다.

### 3. 장기 기억과 과거 회수의 분리
Hermes는 memory와 session_search를 분리해서 씁니다.

- memory는 오래 가는 선호와 규칙
- session_search는 과거 세션과 논의 회수

이 구조 덕분에 모든 과거를 매번 프롬프트에 계속 달고 다니지 않아도 됩니다. 필요할 때 검색해서 다시 끌어오면 됩니다.

### 4. skill로 절차를 외부화
반복 작업이 계속 나오는 운영에서는, 대화 기억보다 절차 기억이 더 중요해집니다. Hermes는 skill로 해결 절차를 외부화할 수 있기 때문에, 같은 작업을 긴 대화 문맥에만 의존하지 않아도 됩니다.

즉 Hermes는 컨텍스트 부패를 “버퍼 관리 문제”로만 보지 않고, **기억의 역할 분리 문제**로 같이 다룹니다.

![Hermes는 무엇으로 긴 운영을 버티나](/assets/build-journey/why-context-rot-happens-and-how-hermes-reduces-it/article-figure-02-hermes-context-strategy.png)

## 그렇다고 Hermes면 자동으로 다 해결되나
여기서 중요한 오해가 하나 있습니다.

아무리 구조가 좋아도, 긴 운영에서는 자동 장치만으로 끝나지 않습니다.

왜냐하면:
- 압축은 결국 손실 압축이고
- 세션이 많아질수록 검색 품질은 기록 품질에 영향을 받고
- 중요 판단이 문서 밖에 없으면 다시 찾아도 애매해지기 때문입니다.

즉 Hermes는 **컨텍스트 부패 대응형 시스템**이지만,
운영자가 정보 아키텍처를 같이 관리해줘야 진짜 강해집니다.

---

## 그래서 운영자는 무엇을 추가로 해야 하나
우리 기준에서는 아래 다섯 가지가 핵심입니다.

### 1. 대화를 소스 오브 트루스로 두지 않는다
대화는 작업 공간입니다. 기준점은 문서여야 합니다.

중요한 결정, 반복 규칙, 운영 철학은 채팅 안에만 두지 말고 위키나 Topic 문서로 올려야 합니다.

### 2. 긴 스레드는 적당한 시점에 handoff 후 새 세션으로 넘긴다
한 스레드 안에서 모든 걸 끝내려 하지 않는 게 좋습니다.

특히:
- 주제가 너무 넓어졌거나
- 압축이 반복됐거나
- 결정과 실행이 여러 번 바뀌었으면

짧은 handoff를 남기고 새 세션으로 넘기는 편이 더 선명합니다.

좋은 handoff는 다섯 줄이면 충분합니다.
- 목표
- 현재 상태
- 핵심 결정
- 남은 리스크
- 다음 액션

### 3. 안정 정보와 작업 중간정보를 분리한다
- 오래 갈 선호와 규칙 → memory
- 반복 절차 → skill
- 프로젝트 사실과 기준 → 문서
- 일회성 사고 과정 → session

이 분리가 안 되면 기억이 섞이고, 사용자는 그걸 성능 저하처럼 느끼게 됩니다.

### 4. 과거 대화는 기억에 기대지 말고 검색으로 회수한다
오래된 맥락을 “분명 알겠지”라고 기대하는 것보다, session_search로 꺼내서 근거를 다시 세우는 편이 훨씬 안전합니다.

### 5. 반복 업무는 skill로 승격한다
반복 작업을 계속 긴 대화 안에서만 풀고 있으면, 컨텍스트를 매번 새로 태웁니다. 검증된 절차는 skill로 고정해야 합니다.

![좋은 운영은 결국 무엇을 습관으로 만드나](/assets/build-journey/why-context-rot-happens-and-how-hermes-reduces-it/article-figure-03-operating-rules.png)

## 우리처럼 오래 쓰는 팀에선 무엇이 가장 중요할까
장기 운영에서는 모델 성능 자체보다 **어디에 무엇을 남기느냐**가 더 중요해집니다.

우리 기준에서 보면:
- 하비 같은 메인 창구는 판단과 통합을 맡고
- 방울이와 뽀동이는 역할별 산출물을 만들고
- 중요한 기준은 Obsidian이나 문서에 올라가고
- 반복 작업은 skill로 굳어지고
- 과거 논의는 session_search로 다시 회수되는 구조가 안정적입니다.

즉 장기 운영에서는 “더 긴 컨텍스트”보다 **더 좋은 경계와 더 좋은 저장 위치**가 더 중요합니다.

---

## 우리가 실제로 세울 만한 운영 원칙
### 원칙 1. 대화는 작업공간이고 문서는 기준점이다.
### 원칙 2. 압축이 반복되기 전에 handoff 후 새 세션으로 넘긴다.
### 원칙 3. 과거 회수는 session_search, 장기 선호는 memory, 반복 절차는 skill로 나눈다.
### 원칙 4. 중요한 결정은 채팅 중간에만 두지 않는다.
### 원칙 5. 길어질수록 모델을 탓하기 전에 정보 구조를 먼저 본다.

---

## 결론
긴 대화와 세션 누적이 많아질수록 에이전트가 흐려지는 건 이상한 일이 아닙니다. 오히려 자연스러운 일입니다.

하지만 그게 곧 “장기 운영은 불가능하다”는 뜻은 아닙니다.
Hermes는 이미 자동 압축, 세션 분리, memory, session_search, skill 같은 장치를 갖고 있어서 일반적인 긴 채팅형 운영보다 훨씬 덜 무너집니다.

핵심은 여기서 한 단계 더 가는 겁니다.
**Hermes의 설계를 믿되, 운영 원칙까지 같이 세워야 한다.**
그때부터 컨텍스트 부패는 막연한 공포가 아니라, 관리 가능한 운영 문제로 바뀝니다.

## FAQ
### 1. 컨텍스트 부패는 결국 피할 수 없는 건가요?
완전히 0으로 만들기는 어렵습니다. 대신 구조와 운영 습관으로 꽤 줄일 수 있습니다. Hermes의 자동 압축·세션 분리·검색 기반 회수는 그걸 위한 장치입니다.

### 2. memory만 잘 쓰면 해결되나요?
아닙니다. memory는 장기 선호와 규칙 저장소입니다. 과거 대화 회수는 session_search가 더 맞고, 반복 절차는 skill이 더 맞습니다.

### 3. 긴 대화가 계속되면 무조건 새 세션으로 갈아타야 하나요?
무조건은 아닙니다. 다만 압축이 반복되거나 주제가 넓어졌다면 handoff 후 새 세션으로 넘기는 편이 품질이 더 좋습니다.

### 4. 우리도 지금 관리가 필요한 단계인가요?
네. 다만 위험 신호라기보다 성장 단계에 가깝습니다. 세션과 자산이 많아질수록 문서화, skill화, 검색 습관을 같이 강화하는 게 좋습니다.
