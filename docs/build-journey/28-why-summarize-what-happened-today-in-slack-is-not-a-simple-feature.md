# Slack에서 오늘 있었던 일 요약해줘가 그냥 되는 기능이 아닌 이유

## TL;DR
많은 사람이 Slack에서 봇에게 “오늘 있었던 일 요약해줘”라고 말하면 그냥 되는 기능처럼 느낍니다. 하지만 실제로는 그렇지 않습니다. 이 기능은 단순한 요약 기능이 아니라, **Slack 데이터를 읽는 도구 + 어떤 기간을 볼지 정하는 규칙 + 어떤 형식으로 정리할지 정하는 스킬**이 함께 있어야 작동합니다. 즉, 이건 문장 생성 문제가 아니라 운영 시스템 문제입니다.

![Slack에서 오늘 있었던 일 요약해줘가 그냥 되는 기능이 아닌 이유](/assets/build-journey/why-summarize-what-happened-today-in-slack-is-not-a-simple-feature/og-card-final.png)

## 답부터 말하면
Slack에서 오늘 있었던 일을 요약하게 하려면, 먼저 그 봇이 **실제 Slack 데이터를 읽을 수 있어야 합니다.**

여기서 가장 많이 생기는 오해는 이겁니다.

“같은 Slack 안에 있으니까 봇도 다 알고 있겠지.”

하지만 실제 시스템은 그렇게 동작하지 않습니다. 봇은 Slack 안에 있다는 이유만으로 자동으로 모든 채널과 스레드 히스토리를 다 아는 것이 아닙니다. 필요한 권한과 도구가 붙어 있어야 하고, 그 도구가 실제 실행 경로에도 살아 있어야 합니다.

즉,
Slack 안에 존재하는 것과 Slack 데이터를 읽을 수 있는 것은 전혀 다른 문제입니다.

## 왜 사람들은 이걸 그냥 되는 기능으로 느낄까
사용자 입장에서는 자연스럽습니다.

- 이미 Slack 안에서 대화 중이다.
- 봇이 바로 앞 메시지에는 답한다.
- 그러면 오늘 있었던 일쯤은 알아서 정리할 수 있을 것 같다.

하지만 시스템은 다층 구조입니다.

봇이 메시지에 대답하는 기능과, 실제 Slack 워크스페이스를 뒤져서 오늘의 중요한 흐름을 수집하는 기능은 서로 다릅니다.

앞자는 “대화 기능”이고,
뒤자는 “데이터 조회 기능”입니다.

이 둘을 구분하지 않으면,
요약 기능이 안 될 때 자꾸 프롬프트만 고치게 됩니다.

![핵심 구조](/assets/build-journey/why-summarize-what-happened-today-in-slack-is-not-a-simple-feature/article-figure-01-why-summarize-what-happened-today-in-slack-is-not-a-simple-feature.png)

## 실제로는 어떤 구성 요소가 필요한가
### 1. Slack 앱 연결
봇이 워크스페이스에 연결돼 있어야 합니다.
그리고 필요한 범위의 채널에 접근 가능해야 합니다.

### 2. Slack 데이터를 읽는 도구
진짜 중요한 건 이 부분입니다.
예를 들면:
- 채널 목록을 보는 도구
- 채널 히스토리를 읽는 도구
- 스레드 답글을 읽는 도구
- 워크스페이스 활동을 넓게 스캔하는 도구

이런 것이 있어야 “오늘 무슨 일이 있었는가”를 실제로 수집할 수 있습니다.

### 3. 요약 스킬
그 다음에야 스킬이 의미를 갖습니다.
스킬은 아래를 담당합니다.
- 오늘/어제/최근 3일 같은 기간 해석
- 중요한 것과 잡음을 나누는 기준
- 최종 결과를 어떤 구조로 보여줄지

즉,
도구는 데이터를 가져오고,
스킬은 그 데이터를 사람에게 이해되게 정리합니다.

## 이번에 우리가 본 세 가지 길
이 문제를 실무적으로 보면 세 가지 길이 있었습니다.

### 첫 번째 길: 세션이나 기억 기반으로 얼추 요약하기
쉽게 보이지만 위험합니다.
왜냐하면 이건 “실제 Slack 오늘치”가 아니라,
에이전트가 기억하거나 저장해 둔 일부 대화를 요약하는 쪽으로 흐르기 쉽기 때문입니다.

겉보기에는 그럴듯해도,
정확한 Slack 브리핑과는 다를 수 있습니다.

### 두 번째 길: Slack API를 직접 별도 코드로 붙이기
유연성은 높습니다.
원하는 만큼 세밀하게 제어할 수 있습니다.
대신 권한, 에러 처리, 포맷, 유지보수를 직접 책임져야 합니다.

### 세 번째 길: Hermes가 제공하는 native Slack tool을 쓰기
우리가 택한 길이 이쪽입니다.
이미 Hermes 안에
- 채널 히스토리
- 스레드 replies
- 채널 목록
- workspace activity
를 다루는 tool이 있었고,
이걸 briefing skill과 연결하는 방식으로 풀었습니다.

이 방식의 장점은,
새 시스템을 0부터 만드는 대신 운영 체계 안으로 녹일 수 있다는 점입니다.

## 왜 이 작업이 생각보다 길어졌나
겉으로 보면 “오늘 있었던 일 요약해줘” 한 문장입니다.
하지만 실제로는 아래를 동시에 맞춰야 했습니다.

- Slack tool이 실제로 등록돼 있는가
- 그 tool이 현재 실행 경로에 실제로 노출되는가
- 채널/스레드 단위로 충분히 조회되는가
- 결과가 단순 메모가 아니라 Slack briefing 형식으로 나오는가
- 정기 크론에서도 같은 기준으로 도는가

즉,
문장 하나 뒤에 시스템 문제가 숨어 있었던 겁니다.

## 여기서 중요한 교훈
가장 중요한 교훈은 이겁니다.

**요약 기능은 글쓰기 기능이 아니라, 먼저 데이터 접근 기능이다.**

이걸 놓치면,
기능이 안 될 때마다 프롬프트만 손보게 됩니다.
하지만 실제 원인은 스킬이 아니라 도구 연결과 실행 경로에 있을 수 있습니다.

## 우리가 운영 원칙으로 얻은 것
이제 기준은 명확해졌습니다.

1. “Slack에서 오늘 있었던 일 요약”은 반드시 live Slack data 기반으로 본다.
2. session_search나 transcript는 보조 참고일 뿐 1차 소스가 아니다.
3. 요약 기능 문제는 프롬프트보다 먼저 도구 연결 문제를 확인한다.
4. 스킬은 도구를 대체하지 않는다. 스킬은 도구 위에서 동작한다.

## 결론
Slack에서 봇에게 “오늘 있었던 일 요약해줘”라고 하는 기능은 그냥 되는 기능이 아닙니다.

그 기능이 자연스럽게 느껴질 정도가 되려면,
사실 뒤에서는 Slack 앱 연결, 실제 조회 도구, 실행 경로, 요약 스킬, 출력 포맷이 다 맞아야 합니다.

즉,
이건 간단한 말 한마디처럼 보이지만,
실제로는 **운영 시스템이 잘 설계됐을 때만 자연스럽게 보이는 기능**입니다.

## FAQ
### 1. Slack 안에서 대화 중이면 봇이 오늘 있었던 일을 다 아는 것 아닌가요?
아닙니다. 대화 가능하다는 것과 실제 워크스페이스 데이터를 읽는 것은 다른 기능입니다.

### 2. 그러면 요약 스킬만 만들면 안 되나요?
안 됩니다. 실제 Slack 데이터를 읽는 도구가 먼저 있어야 하고, 스킬은 그다음 단계입니다.

### 3. 왜 세션 기반 요약으로는 안 되나요?
그건 “Slack 전체 오늘치”가 아니라 에이전트가 접한 일부 대화 회수에 가까워질 수 있기 때문입니다.
