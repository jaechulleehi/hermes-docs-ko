# 자료조사는 왜 검색이 아니라 검증 파이프라인이어야 할까

## TL;DR
AI 에이전트에게 자료조사를 맡길 때 가장 먼저 볼 것은 “무슨 검색 엔진을 쓰는가”가 아닙니다. 더 중요한 것은 **질문 분해 → 후보 확장 → 접근 확인 → 원문 검증 → 근거 등급 → 판단 묶음 → 콘텐츠화**가 한 줄로 이어지는가입니다. DuckDuckGo, SearXNG, API, RSS, blocked-page fallback은 각각 좋은 도구지만, 검색 결과를 곧바로 사실로 올리면 조사는 빨라지는 대신 위험해집니다. 에르메스에서는 검색 도구를 discovery layer로 쓰고, 최종 판단은 원문과 공식 자료에서 다시 확인하는 구조로 운영합니다.

![자료조사 검증 파이프라인](/assets/build-journey/research-verification-pipeline/research-pipeline-main.png)

## 답부터 말하면
자료조사는 검색을 많이 하는 일이 아닙니다.

자료조사는 **믿을 수 있는 판단을 만들기 위해 후보를 찾고, 막힌 곳을 열어 보고, 원문으로 확인하고, 쓸 수 있는 형태로 구조화하는 일**입니다.

그래서 좋은 자료조사 에이전트는 단순히 링크를 많이 가져오는 에이전트가 아닙니다.
좋은 조사 에이전트는 아래를 구분합니다.

- 검색 결과와 확정 사실
- 원문 접근 실패와 자료 부재
- 공식 자료와 2차 해석
- 신규 신호와 기존 중복
- 블로그 후보와 아직 보류할 후보

이 구분이 없으면 결과는 많아지지만 판단은 약해집니다.

## 왜 검색 도구부터 늘리면 위험할까
처음에는 검색 도구를 더 붙이면 조사가 좋아질 것처럼 보입니다.

DuckDuckGo를 붙이고, SearXNG를 붙이고, 여러 쿼리를 병렬로 돌리면 후보는 확실히 많아집니다.
하지만 후보가 많아진다고 근거가 강해지는 것은 아닙니다.

검색 결과에는 보통 이런 문제가 섞입니다.

- 날짜가 최신처럼 보이지만 실제 원문은 오래된 경우
- snippet은 그럴듯하지만 본문이 다른 경우
- 같은 보도자료를 여러 매체가 반복한 경우
- 로그인, 구독, 지역 제한 때문에 본문 확인이 안 되는 경우
- 원문보다 검색 엔진의 요약이 더 강하게 말하는 경우

이 상태에서 바로 “조사 완료”라고 하면, 에이전트는 빠르지만 위험한 보고자가 됩니다.

## 에르메스식 자료조사 흐름
에르메스에서는 자료조사를 한 번에 처리하지 않습니다.
순서는 고정하고, 각 단계의 성공 기준을 다르게 둡니다.

흐름은 이렇게 이어집니다.

1. **질문 분해**  
   무엇을 찾을지보다 먼저, 왜 찾는지와 어떤 판단에 쓸지를 나눕니다.

2. **후보 확장**  
   DuckDuckGo, SearXNG, 공개 API, RSS 같은 discovery layer로 후보를 넓힙니다.

3. **접근 확인**  
   원문이 실제로 열리는지, 렌더링이 필요한지, 로그인이나 구독 장벽이 있는지 확인합니다.

4. **원문 검증**  
   검색 snippet이 아니라 원문, 공식 문서, 공식 데이터, 원문 listing page에서 다시 확인합니다.

5. **근거 등급**  
   공식 자료, 원문, RSS, metadata, mirror를 같은 무게로 보지 않고 증거 강도를 나눕니다.

6. **판단 묶음**  
   링크만 모으지 않고 확정, 후보, 보류를 나눠 다음 사람이 판단할 수 있게 만듭니다.

7. **콘텐츠화**  
   하비가 구조를 잡고, 필요한 경우 뽀동이가 읽히는 글로 재구성합니다.

이 순서가 있어야 검색량이 조사 품질로 바뀝니다.
순서가 없으면 후보 확장과 사실 확정이 섞이고, 에이전트는 빠르지만 위험한 보고자가 됩니다.

### 1단계. 질문을 먼저 쪼갠다
검색어를 넣기 전에 질문을 나눕니다.

예를 들어 “이 주제 조사해줘”라고 하면 바로 검색하지 않습니다.
먼저 아래를 봅니다.

- 시장 신호를 찾는 일인가
- 기술 사실을 확인하는 일인가
- 블로그 소재를 찾는 일인가
- 최신성이 중요한가
- 공식 출처가 필요한가
- 기존 콘텐츠와 중복을 확인해야 하는가

질문을 나누지 않으면 검색 결과는 많아지지만, 무엇을 믿어야 하는지 판단하기 어려워집니다.

### 2단계. discovery layer로 후보를 넓힌다
후보를 넓히는 단계에서는 검색 도구를 씁니다.

여기서 DuckDuckGo와 SearXNG가 의미가 있습니다.

DuckDuckGo 기반 검색은 가볍게 후보를 찾는 데 좋습니다.
SearXNG는 여러 검색 소스를 묶어 더 넓게 훑는 discovery layer로 쓸 수 있습니다.
insane-search류 아이디어는 쿼리 확장, 병렬 검색, 결과 군집화, 중복 제거 패턴을 참고하기 좋습니다.

하지만 이 단계의 결과는 아직 **후보**입니다.
검색 결과는 입구이지 결론이 아닙니다.

### 3단계. 막힌 페이지는 접근 실패부터 분류한다
좋은 후보를 찾았는데 원문이 막히는 경우가 많습니다.

이때 중요한 것은 무작정 우회하는 것이 아니라, 실패의 종류를 분류하는 것입니다.

![막힌 페이지 확인 순서](/assets/build-journey/research-verification-pipeline/research-fallback-ladder.png)

확인 순서는 다음과 같습니다.

1. 원문 페이지 직접 접근
2. 브라우저 렌더링 확인
3. curl과 모바일 user-agent 확인
4. RSS, Atom, sitemap 확인
5. OGP, JSON-LD, 공개 JSON endpoint 확인
6. reader mirror 확인
7. 공식 자료로 재확인

이 순서가 있으면 “안 보인다”와 “검증할 수 없다”를 분리할 수 있습니다.

### 4단계. 최종 판단은 원문과 공식 자료로 한다
검색 도구가 후보를 찾았다면, 판단은 원문에서 해야 합니다.

증거 강도는 보통 이렇게 봅니다.

1. 공식 API, 공식 문서, 공식 changelog, 공식 release note
2. 원문 article, 원문 listing page
3. RSS, Atom, sitemap
4. JSON-LD, OGP, Schema metadata
5. 검색 결과, mirror, archive

검색 결과와 mirror는 도움이 됩니다.
하지만 핵심 주장을 확정하는 기준으로는 약합니다.

그래서 보고할 때도 표현을 바꿉니다.

나쁜 표현은 이렇습니다.

> SearXNG에서 나왔으니 맞습니다.

좋은 표현은 이렇습니다.

> SearXNG로 후보를 넓혔고, 원문과 공식 자료에서 재확인했습니다.

이 작은 차이가 조사 품질을 가릅니다.

### 5단계. 조사 결과를 바로 글로 만들지 않는다
조사 결과가 많다고 좋은 블로그가 되는 것은 아닙니다.

방울이가 조사한 결과는 먼저 판단 가능한 형태로 정리되어야 합니다.

좋은 handoff에는 아래가 들어갑니다.

- 상태: 확정, 후보, 보류
- 왜 중요한가
- 확인된 사실
- 근거 URL과 증거 강도
- 접근 제한
- 기존 콘텐츠와의 중복 여부
- 실무적 의미
- 글감으로 쓸 수 있는 각도
- 다음 확인이 필요한 질문

![조사에서 콘텐츠까지](/assets/build-journey/research-verification-pipeline/research-agent-handoff.png)

그다음 하비가 목차와 판단 구조를 잡습니다.
필요하면 뽀동이가 읽히는 글로 재구성합니다.

즉, 조사는 링크 묶음으로 끝나지 않습니다.
조사는 판단 묶음이 되어야 다음 단계로 넘어갈 수 있습니다.

## SearXNG는 그래서 어디에 쓰나
SearXNG는 검색 결과를 확정하는 도구가 아닙니다.

에르메스에서 SearXNG는 **후보를 넓히는 discovery layer**입니다.

좋은 쓰임은 이렇습니다.

1. 여러 검색 소스에서 후보를 넓힌다.
2. 중복과 낮은 품질 결과를 걷어낸다.
3. 실제 원문이나 공식 소스로 들어간다.
4. 날짜, 제목, 본문, 주장, 출처를 확인한다.
5. 검증된 것만 handoff-ready로 올린다.

이 구조로 쓰면 SearXNG는 강력합니다.
반대로 검색 결과만 보고 확정하면 위험합니다.

## cron에 넣을 때도 같은 원칙이 필요하다
자료조사 cron은 자동 검색기가 되면 안 됩니다.
반복 판단을 자동화해야 합니다.

시장 조사 cron이라면 아래가 중요합니다.

- watchlist 기반 공식 source 우선 확인
- discovery query 확장
- 신규 후보와 기존 후보 중복 제거
- 차단 페이지 fallback
- 확인된 신호만 후보 backlog에 저장

뉴스레터 cron이라면 아래가 중요합니다.

- 메일함 원문 기반 수집
- 직전 성공 이후 항목 우선
- 중복 최소화
- 링크와 발행일 검증
- 콘텐츠 후보, 운영 참고, 버릴 항목 분리

자동화의 목표는 “많이 긁기”가 아니라 “다시 볼 가치가 있는 것만 남기기”입니다.

## 에르메스에서 역할은 어떻게 나뉘나
이 구조는 에이전트 역할 분리와 잘 맞습니다.

방울이는 후보를 넓히고 검증합니다.
하비는 판단 구조를 만들고 최종 품질을 책임집니다.
뽀동이는 검증된 재료를 읽히는 콘텐츠로 바꿉니다.

역할을 나누는 이유는 단순합니다.
자료조사는 넓게 봐야 하고, 글은 좁혀야 하기 때문입니다.

한 에이전트가 처음부터 끝까지 모두 처리하면, 넓히기와 좁히기가 자주 섞입니다.
그 결과 조사도 애매하고 글도 애매해집니다.

## 이번 작업에서 얻은 운영 기준
에르메스 자료조사 기준은 이렇게 정리할 수 있습니다.

1. discovery와 verification을 분리한다.
2. 검색 결과는 근거가 아니라 입구다.
3. 막힌 페이지는 접근 실패인지, 자료 부재인지, 유료 장벽인지 먼저 분류한다.
4. 공식 API와 RSS가 있으면 먼저 쓴다.
5. handoff는 링크 묶음이 아니라 판단 묶음이어야 한다.
6. 반복되는 조사 패턴은 skill로 만들고 cron에 태운다.
7. 최종 구조와 품질 책임은 하비가 진다.

## 결론
AI 에이전트의 자료조사 역량은 검색 도구 개수로 결정되지 않습니다.

진짜 차이는 검색 뒤에 있습니다.
후보를 어떻게 검증하는지, 막힌 곳을 어떻게 분류하는지, 확인된 사실과 보류할 후보를 어떻게 나누는지, 그리고 그 결과를 어떻게 다음 작업자가 쓸 수 있게 넘기는지가 중요합니다.

자료조사는 검색이 아니라 검증 파이프라인입니다.
그 파이프라인이 있어야 에이전트는 빠른 검색자가 아니라 믿을 수 있는 조사자가 됩니다.

## FAQ
### 1. DuckDuckGo와 SearXNG 중 무엇을 먼저 써야 하나요?
가볍게 후보를 찾을 때는 DuckDuckGo가 편합니다. 더 넓게 후보를 모으거나 여러 검색 소스를 묶고 싶을 때는 SearXNG가 좋습니다. 다만 둘 다 최종 근거가 아니라 discovery layer로 써야 합니다.

### 2. 검색 결과만으로도 충분한 경우가 있나요?
단순한 후보 수집에는 충분할 수 있습니다. 하지만 날짜, 수치, 정책, 제품 변경, 시장 신호처럼 판단이 필요한 내용은 원문이나 공식 자료로 다시 확인해야 합니다.

### 3. 차단된 페이지는 reader mirror를 쓰면 끝인가요?
아닙니다. reader mirror는 보조 접근 경로입니다. 가능하면 원문, 공식 문서, RSS, 구조화 데이터로 다시 확인해야 합니다.
