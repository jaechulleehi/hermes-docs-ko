# 아키텍처

Hermes Agent의 내부 구조를 설명합니다.

## 시스템 개요

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│   CLI   │  │ Gateway │  │   ACP   │
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     └────────────┼────────────┘
                  │
           ┌──────▼──────┐
           │   AIAgent   │
           └──────┬──────┘
                  │
     ┌────────────┼────────────┐
     │            │            │
┌────▼────┐ ┌────▼────┐ ┌────▼────┐
│ Prompt  │ │Provider │ │  Tool   │
│ Builder │ │ Resolve │ │Dispatch │
└─────────┘ └─────────┘ └─────────┘
```

3개의 진입점(CLI, Gateway, ACP)이 모두 AIAgent로 흘러들어가며, AIAgent가 프롬프트 구성, 제공자 선택, 도구 실행을 조율합니다.

## 디렉토리 구조

```
hermes-agent/
├── run_agent.py          # 에이전트 실행 진입점
├── cli.py                # CLI 진입점
├── model_tools.py        # 모델 도구 정의
├── agent/                # 핵심 에이전트 로직
│   ├── ai_agent.py       # AIAgent 클래스
│   ├── prompt_builder.py # 프롬프트 구성
│   ├── provider.py       # 제공자 관리
│   └── session.py        # 세션 관리
├── hermes_cli/           # CLI 구현
├── tools/                # 40+ 도구
├── gateway/              # 메시징 게이트웨이
├── skills/               # 스킬 시스템
├── acp_adapter/          # ACP 통합
└── tests/                # 테스트
```

## 데이터 흐름

### CLI 흐름

```
사용자 입력 → cli.py → AIAgent → 프롬프트 구성 → LLM 호출
     → 도구 실행 → 결과 반환 → 응답 표시
```

### Gateway 흐름

```
메시지 수신 → 플랫폼 어댑터 → 세션 라우터 → AIAgent
     → 처리 → 응답 → 플랫폼 어댑터 → 메시지 전송
```

### Cron 흐름

```
스케줄러 트리거 → 크론 작업 로드 → AIAgent → 처리 → 결과 저장/알림
```

## 주요 서브시스템

### 1. 에이전트 루프

AIAgent의 핵심 반복 루프:

1. 프롬프트 구성
2. LLM 호출
3. 응답 파싱 (텍스트 / 도구 호출)
4. 도구 실행
5. 결과를 컨텍스트에 추가
6. 반복 또는 완료

### 2. 프롬프트 시스템

시스템 프롬프트 구성 순서:

1. SOUL.md (성격)
2. 컨텍스트 파일 (HERMES.md, AGENTS.md)
3. 도구 정의
4. 메모리 (MEMORY.md, USER.md)
5. 세션 히스토리

### 3. 제공자 해석

16개 이상의 AI 제공자를 지원하며, 모델명으로 자동 라우팅합니다:

```
"anthropic/claude-sonnet-4" → Anthropic API
"openai/gpt-4o" → OpenAI API
"google/gemini-2.0-flash" → Google API
"ollama/llama3" → 로컬 Ollama
```

### 4. 도구 시스템

- 도구 정의 → 도구셋 그룹핑 → 플랫폼별 필터링
- MCP 도구 통합 (접두사 자동 부여)
- 위험 명령 감지 및 승인

### 5. 세션 영속성

SQLite 기반 세션 저장:
- 대화 기록
- 도구 실행 결과
- 메타데이터

### 6. 플러그인 시스템

외부 메모리 제공자, 도구, 채널 어댑터 등을 플러그인으로 확장할 수 있습니다.

## 설계 원칙

| 원칙 | 설명 |
|------|------|
| **프롬프트 안정성** | 시스템 프롬프트 구조를 일관되게 유지 |
| **관찰 가능한 실행** | 모든 도구 호출과 결과를 로그로 추적 |
| **중단 가능** | 언제든 사용자가 작업을 중단 가능 |
| **플랫폼 무관 코어** | 핵심 로직은 플랫폼에 의존하지 않음 |
| **느슨한 결합** | 서브시스템 간 최소한의 의존성 |
| **프로필 격리** | 여러 프로필이 독립적으로 동작 |

10. 전체 아키텍처 복습
