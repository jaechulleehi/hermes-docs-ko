# 자료조사 검증 파이프라인 운영

Hermes에서 자료조사를 맡길 때 효과가 컸던 방식은 검색 도구를 하나 더 붙이는 것이 아니라 **조사 단계를 나누고 각 단계의 성공 기준을 다르게 둔 것**입니다.

## 이 사례의 답
- 검색 도구는 후보를 넓히는 discovery layer다.
- 최종 판단은 원문, 공식 API, RSS, 구조화 데이터로 다시 확인한다.
- 막힌 페이지는 접근 실패와 검증 실패를 분리해서 다룬다.
- 조사 결과는 링크 묶음이 아니라 판단 묶음으로 handoff한다.

## 어떤 상황에서 썼나
이 구조는 아래 상황에서 특히 유효했습니다.

- 시장 조사 cron에서 신규 신호와 중복 신호를 나눠야 할 때
- 뉴스레터 cron에서 메일함 항목을 콘텐츠 후보와 단순 참고로 나눠야 할 때
- 차단된 기사나 JS 기반 페이지를 확인해야 할 때
- 방울이 조사 결과를 뽀동이 콘텐츠 작업으로 넘겨야 할 때
- 검색 snippet과 원문 사실이 섞이는 것을 막아야 할 때

## 실제 운영 흐름
1. 하비가 질문을 분해하고 조사 목적을 정한다.
2. 방울이가 DuckDuckGo, SearXNG, API, RSS 등으로 후보를 넓힌다.
3. 막힌 페이지는 fallback ladder로 접근 가능성을 확인한다.
4. 원문과 공식 자료 기준으로 확정, 후보, 보류를 나눈다.
5. 하비가 목차와 판단 구조를 만든다.
6. 필요한 경우 뽀동이가 공개 글이나 보고서로 재구성한다.
7. 반복되는 흐름은 skill과 cron으로 고정한다.

## 왜 이 방식이 좋았나
### 1. 검색 결과와 사실을 분리할 수 있었다
검색 결과는 빠르지만 자주 과감합니다. 검증 단계를 분리하니 그럴듯한 결과를 사실처럼 보고하는 일이 줄었습니다.

### 2. 차단된 페이지를 더 침착하게 다룰 수 있었다
안 보이는 페이지가 나와도 바로 포기하지 않고, 원문 접근, 브라우저 렌더링, RSS, metadata, 공개 endpoint 순서로 확인할 수 있었습니다.

### 3. 콘텐츠 품질이 좋아졌다
조사 결과가 상태, 근거, 한계, 각도로 정리되니 블로그나 강의로 넘길 때 재작업이 줄었습니다.

## 이 사례에서 얻은 운영 원칙
- discovery와 verification을 같은 단계로 섞지 않는다.
- 공식 자료가 있으면 검색 결과보다 먼저 본다.
- blocked page는 실패 원인을 먼저 분류한다.
- handoff에는 근거 강도와 접근 제한을 반드시 남긴다.
- cron은 많이 긁는 장치가 아니라 좋은 후보만 남기는 장치여야 한다.
