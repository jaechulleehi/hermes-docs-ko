# 팀 내 메인 창구 운영

Hermes를 팀 내 업무에 붙일 때 가장 먼저 효과가 났던 활용 방식은 **사용자는 하비 하나만 부르고, 내부에서는 조사·정리·통합을 분리하는 구조**였습니다.

## 이 사례의 답
- 사용자는 누구를 불러야 할지 고민하지 않아도 된다.
- 방울이는 조사, 뽀동이는 정리, 하비는 판단과 통합을 맡는다.
- 최종 결과물의 형식과 품질 책임은 하비가 끝까지 진다.

즉, 이 사례의 핵심은 멀티에이전트를 많이 드러내는 것이 아니라 **사용자 경험은 단순하게, 내부 운영은 더 정교하게 만드는 것**입니다.

## 어떤 상황에서 썼나
다음 같은 상황에서 이 구조가 특히 유효했습니다.

- 요청이 짧게 들어오지만 뒤에서는 조사와 정리가 같이 필요한 경우
- Slack 같은 채널에서 사용자가 에이전트 구조를 직접 관리하고 싶어 하지 않는 경우
- 중간 산출물보다 최종 보고 형태가 더 중요한 경우
- 여러 에이전트를 쓰되, 누가 최종 책임자인지는 선명해야 하는 경우

## 실제 운영 흐름
1. 사용자는 하비에게만 요청한다.
2. 하비가 요청을 해석한다.
3. 조사형 작업이면 방울이를 태운다.
4. 문서화/초안화가 필요하면 뽀동이를 태운다.
5. 하비가 결과를 검수하고 충돌을 조정한다.
6. 최종안만 사용자에게 전달한다.

## 왜 이 방식이 좋았나
### 1. 사용자가 구조를 공부하지 않아도 된다
사용자는 "이건 누구에게 물어야 하지?"를 덜 고민하게 됩니다. 진입 장벽이 낮아지고 follow-up도 자연스러워집니다.

### 2. 최종 책임이 선명하다
조사와 정리는 내부 역할이지만, 최종 결과의 품질 책임은 하비에게 남습니다. 그래서 중간 산출물이 그대로 튀어나오는 일이 줄어듭니다.

### 3. 운영 규칙을 한 점에 모으기 쉽다
언제 직접 처리할지, 언제 위임할지, 어떤 형식으로 보고할지 같은 규칙을 하비 기준으로 정리할 수 있습니다.

## 실제로 조심한 점
### 1. 하비가 병목이 되지 않게 해야 한다
모든 게 하비를 거치기 때문에, 위임 기준과 검수 기준이 흐리면 전체 속도가 느려집니다.

### 2. 위임만 하고 통합을 안 하면 안 된다
조사 결과를 그대로 던지거나 정리 초안을 검수 없이 넘기면 사용자가 다시 편집자가 됩니다.

### 3. 역할 경계가 흐려지면 장점이 사라진다
하비도 조사하고 방울이도 정리하고 뽀동이도 판단까지 해버리면 구조의 강점이 약해집니다.

## 이 사례에서 얻은 운영 원칙
- 사용자는 하비만 부르면 된다.
- 내부 라우팅은 하비가 책임진다.
- 위임해도 최종 통합 책임은 하비에게 남는다.
- 병목이 생기면 사람보다 흐름 설계를 먼저 본다.
