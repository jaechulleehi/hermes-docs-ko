# 위키에서 블로그와 강의로 파생하는 콘텐츠 시스템

Hermes 관련 기록을 운영하면서 두 번째로 효과가 컸던 활용 방식은 **조사 결과와 시행착오를 먼저 위키 원천 문서에 쌓고, 그다음 블로그와 강의로 파생하는 구조**였습니다.

## 이 사례의 답
- 조사 결과는 그대로 공개 콘텐츠가 되지 않는다.
- 먼저 Topic 형태의 원천 문서에서 질문, 기준, 패턴, FAQ 후보를 정리해야 한다.
- 블로그는 질문형/answer-first 구조로 다시 재배열하고, 강의는 전달 순서와 사례 중심으로 다시 짠다.
- 반응과 수정 결과는 다시 위키로 환류한다.

즉, 핵심은 콘텐츠를 매번 새로 쓰는 것이 아니라 **원천 문서를 여러 번 다른 포맷으로 살아남게 만드는 것**입니다.

## 어떤 상황에서 썼나
이 구조는 아래 같은 상황에서 특히 유효했습니다.

- 조사 메모는 많은데 블로그로 옮기면 힘이 빠지는 경우
- 비슷한 내용을 강의와 문서에서 반복해서 다시 써야 하는 경우
- 운영 지식이 흩어져 있어 재사용이 어려운 경우
- 하나의 경험을 위키·블로그·FAQ·강의로 확장하고 싶은 경우

## 실제 운영 흐름
1. 조사 결과와 운영 메모를 모은다.
2. Topic 원천 문서에서 핵심 질문과 lesson learned를 정리한다.
3. 블로그는 질문형 제목, TL;DR, answer-first, FAQ 구조로 재구성한다.
4. 강의는 45분 흐름, 사례, 데모 포인트 기준으로 다시 짠다.
5. 공개 후 반응과 추가 질문을 다시 Topic으로 환류한다.

## 왜 이 방식이 좋았나
### 1. 같은 내용을 처음부터 다시 쓰는 일이 줄었다
위키에 기준점이 남아 있으니, 블로그나 강의를 만들 때 맥락을 다시 복원하는 비용이 크게 줄었습니다.

### 2. 깊이와 전달력을 분리해 다룰 수 있었다
원천 문서는 깊이를 담당하고, 블로그/강의는 전달력을 담당합니다. 그래서 한 문서에 모든 걸 우겨 넣지 않아도 됩니다.

### 3. 콘텐츠가 자산으로 누적된다
한 번 정리한 위키 문서는 FAQ, 체크리스트, 블로그, 강의안으로 계속 파생될 수 있습니다.

## 실제로 조심한 점
### 1. 조사 결과를 그대로 내보내면 안 된다
근거가 많다고 바로 읽히는 글이 되지는 않습니다. 중간에서 질문과 우선순위를 다시 잡아야 합니다.

### 2. 블로그와 강의는 요구하는 구조가 다르다
블로그는 읽히게, 강의는 따라오게 다시 짜야 합니다. 같은 내용이라도 형식이 달라집니다.

### 3. 환류가 빠지면 위키가 금방 죽는다
공개 후 나온 반응과 추가 질문을 다시 Topic에 붙이지 않으면 원천 문서의 가치가 빠르게 떨어집니다.

## 이 사례에서 얻은 운영 원칙
- 깊이는 원천 문서가 만든다.
- 전달력은 재구성이 만든다.
- 좋은 조사 결과는 좋은 출발점이지, 자동으로 좋은 콘텐츠는 아니다.
- 위키는 기록 저장소가 아니라 콘텐츠 모체다.
